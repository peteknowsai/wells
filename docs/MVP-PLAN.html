Phase A — Mature management DONE 2026-05-12 · v0.2.0

  • Pressure controller (~150-200 lines, lib/memoryChunks.ts): inflate balloon on cell start, reclaim on /sleep, deflate on grant requests, track chunks granted / peak / denials.

C.3 — Metrics + warnings

  • Log chunks_granted over time, exposed via well info + a new endpoint.
  • Warning thresholds: p95 capacity >85%, momentary full chunk pool, grant denials.

C.4 — Smoke + tuning

  • Smoke test: spin up cells past the alive ceiling at the old static allocation. Verify grants + reclaims, no OOM.
  • Tune chunk size: confirm 512 MB vs 256 MB / 1 GB.

Phase D — Multi-Lab Colony 1.x · v0.3.0

The Colony layer from docs/naming.md made real. A single Mac (one Lab) hits a hard ceiling around 100-200 cells alive; past that, you add another Mac. This phase makes wells span multiple local-network Macs as one Colony.

Crucial reframe 2026-05-07: the original Phase E was "Linux hosting" — port to KVM-on-Hetzner-VPS. Pete's call: that's not the right move. Cloud VPS breaks the local-first thesis. The path to scale is more local Macs, not cloud. A second Mac in your closet is a Lab; together they're a Colony.

D.1 — Multi-Lab discovery

  • Lab registry: ~/.wells/labs.json lists known Labs. CLI: well lab add|list|remove.
  • Cross-Lab CLI: well create my-cell --lab=mac-mini-2 creates the cell on the named Lab via that Lab's welld API.

D.2 — Lab-to-Lab cell migration

  • Cell migration via Frozen tier: hibernation file → R2, original destroyed, target Lab pulls + resumes. Identity preserved.
  • well move <cell> --to=<lab> CLI orchestrates the migration end-to-end.

D.3 — Colony-aware routing

  • Cells's CF Worker bridge points at a Colony, not a single Lab. Routes to whichever Lab currently hosts the cell.
  • Pulse runs once per Colony, scheduling wakes across all Labs.

D.4 — Multi-Lab pressure balancing (uses Phase C chunks)

  • Cross-Lab memory pressure: migrate a hibernating cell from a full Lab to one with headroom. (Optional / advanced.)
  • Capacity reporting: well colony status shows per-Lab and Colony-wide alive cells, chunk usage, durable counts.

Phase E — Cloud hosting deprioritized, possibly never

Originally "port wells to KVM-on-Hetzner-VPS." Deprioritized 2026-05-07: cloud hosting breaks the cooperation-first economics (metered RAM = paused cells aren't free) and adds latency that defeats sub-millisecond pause/resume.

If we ever do this, it's probably as cold-storage offload for Frozen-tier cells (R2 already covers that) or as a fallback for users without local hardware. Status: non-goal for the foreseeable future. The archived Phase E plan (engine boundary cleanup, Firecracker engine, hosting smoke, Linux docs) lives in git history — reconstitute from there if priorities change.

Loop discipline

When the /mvp-wells loop fires, the running agent should:

  1. Read this file. Find the first phase with unchecked items.
  2. Identify the smallest next checkbox to make progress on.
  3. Implement it. Write tests where it makes sense. Run them.
  4. Commit with a clear message naming the phase + checkbox.
  5. Update the checkboxes in this file (class="task open" → class="task done", swap the ☐ intent). Commit the doc change too.
  6. If a whole phase is complete, mark it done in the status table, squash-merge the phase branch to main, tag the version, then start the next phase.

Bounded: one loop run = roughly one focused chunk of work. If a checkbox is huge, decompose into sub-checkboxes during the run; check the easy ones, defer the rest.

Branch convention: one feature branch per phase. Don't commit to main directly — only the phase-end squash-merge lands on main.

When stuck: write the blocker to docs/BLOCKED.html (date, what was tried, what's needed from Pete). Commit. Stop the run.
