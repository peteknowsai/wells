Introduction

About This Book

Every system grows a dashboard. Mostly they grow into wallpaper — a wall of dials nobody glances at, full of numbers no one knows what to do with. The Wells Dashboard wants to be the opposite of that: a five-second read that tells you the truth, written by an agent who lives inside the thing it's narrating.

This book is the plan for building it. We've already locked the visual design (it's in docs/proposals/dashboard-mock/), already written a build spec (it's in docs/proposals/wells-dashboard-build-spec.html), and already birthed the cell that will host the app. What's left is the actual app. This book lays out the shape of it, what each page does, where the data comes from, and the small architectural decisions worth getting right before the first commit.

🧵
Remember

This is a wells dashboard — the substrate's view of itself. Wells watches the box from the outside: RAM, lifecycle, state, IPs. Whether an agent inside a well is doing useful work is cells's business, and cells gets its own dashboard at cells.cells.md.

Foolish Assumptions

This book assumes you:

  • Know that wells is a self-hosted VM control plane — a daemon (welld) and a CLI (well) that manage stateful Linux VMs on your own hardware.
  • Know that cells is the agent fleet that runs on top of wells — each cell is a long-lived AI agent (Claude Code, Codex, Pi, or whatever harness you like) running inside one of those VMs.
  • Have seen a developer dashboard before — and have opinions about why most of them are too noisy to be useful.
  • Are not opposed to a dashboard that writes prose at you. Even if the agent is occasionally goofy. Especially then.

Part IWhat This Even Is (And Why You'd Want One)

Chapter 1: The Two Jobs of a Dashboard

In This Chapter
  • The dashboard tells you how it's going AND explains what "it" is
  • Why "an IT dashboard" is the wrong reflex
  • Comprehension as a first-class feature

Most dashboards do exactly one job: they tell you operational state. CPU is at X. Memory at Y. Requests-per-second at Z. Useful, in the same way a car's instrument cluster is useful — assuming you already know what a car is.

The wells dashboard is asked to do two jobs at once, every glance:

  • Tell you how things are going — operational status, the Mac's memory, the well-house, what changed lately.
  • Explain the system — the page's structure is the explanation. Every visit re-teaches what wells is and how the pieces fit.

That second job is unusual, and it's the whole reason this dashboard isn't just "a Grafana for wells." Wells is young, and the operator (you, mostly) doesn't always remember every detail at 2 AM. A page that quietly re-teaches the model every time you look at it is worth ten dashboards that assume you already know.

🧵
Remember

Comprehension is a feature. If you're tempted to remove a label because "the operator already knows," put it back. The dashboard's audience includes yourself at 4 AM, six months from now, after a long week.

Chapter 2: Not An IT Dashboard

The current dashboard at lib/dashboard.ts — the one served by renderDashboardHtml() — is exactly the thing we are not building. It's a competent-enough operational view: counts, IPs, statuses, the welld log tail. It's also the developer-reflex default: data in, table out. We're deleting it.

The new dashboard refuses, on purpose:

  • No gauges, dials, or metrics walls. A page covered in numbers is a page that requires interpretation. Make the agent interpret.
  • No raw log tails. "POST /v1/wells was called" is not a thing to show a human. "A well was born" is.
  • No time-series graphs. The dashboard's time horizon is "right now, plus lately." History lives in SQLite for the agent to consult, not for you to squint at.
  • No 12-panel grid. One column, glance-first, depth one click away. The reader's attention is a budget; don't blow it on chrome.
⚠️
Warning

The pull toward "but a small graph would be helpful here" is constant and strong. Resist it. If a graph would add something the narrator's prose doesn't, the prose needs to be better — not the page busier.

Chapter 3: The Wells/Cells Boundary

Wells and cells are two layers of the same stack: wells is the substrate (the VMs), cells is the fleet that lives in those VMs (the agents). They have separate dashboards on purpose.

This dashboard shows…This dashboard does NOT show…
Well lifecycle (born / running / hibernating / stopped / destroyed) Agent reasoning, conversations, tool calls
Host RAM, per-well memory held, the Mac's headroom Cells's pool refill loop, eviction decisions, "warm vs cold"
The image library and where each well was forged from Per-cell DNA / extensions / model chain
Substrate health (welld up? lume respawning? leases sane?) Why an agent decided to scale up or down
State changes ("a well was sealed in 3.8s") "POST /v1/wells/foo/exec returned 200 in 412ms" × 1,000 rows
🧵
Remember

Report the membrane, not the motive. "Three wells born this hour" is a wells fact. "Cells scaled up because traffic spiked" is a cells fact — don't guess at it from the substrate side.

Sidebar — Why no caller attribution

You might wish the dashboard could distinguish "cells made this well" from "you made this well." It can't, and we're not going to make it try. welld authenticates a single shared bearer token (~/.wells/token) and has no notion of "which client." Plumbing per-caller identity through every endpoint for one feature isn't worth it. If you want to know who acted, look at the cells dashboard.

Part IIThe Stack Behind the Glass

Chapter 4: A Cell Named Wells (The Slightly Recursive Bit)

In This Chapter
  • Why the dashboard lives on a cell, not on welld
  • What it means to host a wells dashboard on the wells substrate
  • The bare degradation path that makes the recursion safe

The dashboard lives in a cell. Specifically, a cell named wells, born from the same warm-egg pool as every other cell, with the Pi harness, gpt-5.5 on medium thinking. That cell hosts the Next.js app, the narrator loop, the SQLite database, and the Pi RPC process. The browser URL is https://wells.cells.md. Cells's own dashboard, by the same logic, lives at cells.cells.md.

Pause on that for a moment: the wells dashboard is a cell running on the wells substrate it's watching. It hibernates when nobody's looking. It wakes when you visit the URL. When the dashboard goes to sleep, the substrate it monitors is monitoring it back. It will show up in its own portrait — "11 running eggs, 1 hibernating" — and the hibernating one will be itself.

💡
Tip

This isn't a gimmick. It's the cleanest deployment: the dashboard runs on the exact platform whose value it advertises. If running a Next.js app on a wells cell doesn't work well enough for the dashboard, that's the signal to fix wells — not to move the dashboard.

⚠️
Warning

If the cell crashes, the rich dashboard goes dark. That's why welld itself keeps a bare HTML page at /up — minimal, no agent, no cell. "Substrate is up. N wells. welld vX. Uptime Yh." The always-on truth path. The rich experience is additive.

Technical Stuff

The Next.js app and the narrator loop are both long-lived processes in the cell. The Next.js server hosts the four pages and a single API route (/api/talk) that drives the Pi RPC child process. The narrator loop is a separate node process — it's its own service unit (or just a backgrounded node process under systemd / a tmux session, depending on how Pi-on-a-cell is configured). Two processes, one SQLite file, one Pi agent definition shared as the source of voice and tools.

Chapter 6: Why It Lives On The Cell (And Not Somewhere Sensible Like Vercel)

Reasonable question: why host the Next.js app on a tiny Linux VM with traffic-driven hibernation, rather than slap it on a real PaaS? Three reasons, in order of importance:

  1. The talk page wants local IPC. The chat surface drives a Pi RPC process — the live narrator with tools. Local-on-the-cell, that's a child process the API route talks to over stdin/stdout. Hosted elsewhere, you have to bridge the chat session across a network, authenticate the bridge, deal with reconnects when the cell hibernates. Local wins by a mile.
  2. SQLite wants to be next to its writer. The narrator loop writes SQLite. The Next.js app reads it. On the same filesystem, that's two processes against one file — the design SQLite was built for. Cross-network it becomes "now we need LiteFS or Turso." That's scope we don't want.
  3. Hibernation is a feature. The wells dashboard is a glance-first surface. Nobody's looking → cell sleeps → costs nothing. You load the URL → wake-on-traffic → ~1s wake → page renders from the last SQLite snapshot the narrator wrote before sleeping. The product literally demonstrates the substrate's value while serving its own purpose.
🧵
Remember

The dashboard being on the cell does not mean wells's own health depends on a cell. The bare /up page is served by welld directly, independent of any cell, agent, or app. Lose the cell and you lose the rich experience; you do not lose the substrate's ability to say "I'm up."

Chapter 7: What welld Tells Us (And What It Should)

In This Chapter

  • The four gaps to close on the WELLD track

Welld already exposes GET /dashboard/data as a public, no-auth JSON endpoint. Today's shape: 

{
  generated_at: string,
  daemon: {
    version, started_at, uptime_seconds, degraded,
    lume: { base_url, owned, respawns_last_hour,
            respawns_last_5min, respawns_last_1min },
    vz_xpc_count: number
  },
  wells: [
    { name, status: "running"|"stopped"|"missing",
      ip, created_at, last_running_at }
  ],
  vmnet_leases: { total, orphan_count,
                  orphans: [{ name, ip }] },
  images: [
    { name, aliases, size_bytes, created_at,
      from_well, rinsed, firstboot_supports_static_ip }
  ],
  events: string[]
}

Plenty to start with. Four gaps to close on the WELLD track:

AddWhy
host: { memory_total_bytes, memory_used_bytes } "Memory on the Mini" is the dashboard's hero stat. Today we have zero host-memory exposure — the mockup labels it "estimated" because the total isn't there yet. Probe vm_stat / sysctl from welld.
Per-well memory_resident_bytes Show what each well actually holds, not just its 1 GB allocation. Probe the per-VM Virtualization.VirtualMachine process RSS.
Per-well runtime_state /dashboard/data only knows running|stopped|missing. The real machine has more — hibernating, alive_paused — and the dashboard wants to group by that.
timings — recent op durations "Last seal ran in 3.8 seconds" is in the welld log already (as a structured event). Expose it as a first-class array of recent op timings so the narrator can talk about them without scraping log lines.
💡
Tip

When data is missing, say so — don't fake it. The mockups use a small italic "estimated" or "host total not yet reported by welld" tag. That's the honest move and it's part of the dashboard's voice.

Chapter 8: The SQLite Timeline

SQLite is where the dashboard's memory lives. Every welld poll becomes a snapshot row. Every narrator pass becomes a read row. Lifecycle events get lifted out of the log tail and turned into structured events. Per-well history accumulates.

Starting schema (adjust as the build reveals needs):

snapshots        -- the timeline (one row per welld poll)
  id, polled_at, raw_json,
  well_count, running_count, stopped_count, hibernating_count,
  committed_mb, host_total_mb, degraded

reads            -- the read history (one row per narrator pass)
  id, written_at, body, based_on_snapshot_id

events           -- the "lately" timeline, narrated
  id, occurred_at, kind, summary, well_name?, snapshot_id

wells            -- known wells, upserted each poll
  name, status, runtime_state, ip, cpu, memory_mb, disk_size_gb,
  disk_used_bytes, memory_resident_bytes, uuid, url,
  auto_sleep_seconds, created_at, first_seen, last_seen

well_timeline    -- per-well lifecycle events (for the detail page)
  id, well_name, occurred_at, kind, summary

images           -- snapshot of the image library
  name, aliases, size_bytes, origin, created_at,
  r2_status, wells_forged

metrics          -- timed operations (once welld exposes timings)
  id, op, well_name, duration_ms, occurred_at
🤓
Technical Stuff

The DB file lives at state/wells.db inside the cell (gitignored). Two processes touch it: the narrator loop writes; the Next.js app reads. Use WAL mode (PRAGMA journal_mode=WAL) so the reader never blocks the writer. Concurrency on a single SQLite file with one writer and one reader is a solved problem — don't reach for anything fancier.

Chapter 9: The Narrator Loop (The Heartbeat)

The narrator loop is the engine that keeps "the read" up to date. Each tick:

  1. Poll welld. GET /dashboard/data, plus per-well details for anything missing sizing. Save a snapshots row.
  2. Diff. Compare against the previous snapshot — new wells, state changes, lease changes, image changes, degraded flips, fresh log-event lines.
  3. Decide. Run a Pi pass if something meaningful changed, or it's been ≥ N minutes since the last read (starting cadence: ~3–5 min quiet, immediate on change).
  4. Run Pi. Invoke Pi (print mode or via the SDK) with: the current snapshot, the diff, and the last read. The agent returns a new read (prose) and any new lately events (structured). The agent's voice and rules live in SYSTEM.md.
  5. Write SQLite. Insert reads, events; upsert wells / well_timeline / images / metrics.
  6. Sleep. Until the next tick, or a wake signal. The cell itself may hibernate between ticks — that's fine; the next visitor wakes it.
🧵
Remember

The agent runs on change or on cadence, not on every poll. Most polls return "nothing new" and don't warrant a Pi call. The narrator's voice should feel attentive but not chatty — like a thoughtful colleague glancing over your shoulder, not a junior dev tailing the log.

The transcript is laid out as a sober chat (you on the right, agent on the left). Tool runs appear as action blocks — a small framed box showing what command the agent ran and what the result was — which is the chatbot-vs-agent line we want to draw. The agent isn't "just talking." It did something. You can see what.

🤓
Technical Stuff

The implementation builds on Pi's first-party web-UI component library (@mariozechner/pi-web-ui). It ships a ChatPanel component that handles streaming + tool execution rendering natively. There's a Lit-in-React seam to be aware of — pi-web-ui is Lit-based, our app is Next.js. Three landing options: use <ChatPanel> as a custom element in the React app; make /talk a small standalone Lit app linked from the nav; or skip pi-web-ui and drive pi --mode rpc directly behind a Next.js API route. Decide during build — option (b) is closest to the upstream example.

⚠️
Warning

If you go the RPC route (option c above), do not use Node's readline to split the JSONL stream. It splits on U+2028/2029 and breaks Pi's protocol. Roll your own \n-only splitter.

Part VHow It Talks Back

Chapter 14: The Narrator's Voice

In This Chapter
  • What the narrator does (and what it deliberately doesn't)
  • The voice rules — calm, plain, honest about gaps
  • Why SYSTEM.md is load-bearing

The narrator is a Pi agent. Its job is to look at welld's facts, the recent history, and the current diff, and write one paragraph (the read) that captures what's going on. Every page on the dashboard has a read at the top; they're all written by the same agent following the same rules.

The voice rules (which live in SYSTEM.md on the cell):

  • Operational first, vibes second. Lead with the bottom line — "everything's operational" or "two things worth a glance" — before detail.
  • Plain English, no jargon you wouldn't say out loud. "The Mini's barely touched" beats "host memory utilization is at 16%."
  • Honest about gaps. If welld doesn't expose host RAM total yet, the narrator says so ("I can't tell you exactly — welld doesn't report that yet") rather than guessing.
  • Flag, don't alarm. Stale leases are "worth a glance," not "WARNING." The dashboard never cries wolf.
  • Never quote raw API calls. "POST /v1/wells succeeded" is wallpaper. "A well was born" is a fact.
  • Respect the boundary. Don't guess at why cells made a decision. That's not the narrator's domain.
💡
Tip

Tune the voice by editing SYSTEM.md on the cell, not by fiddling with the loop or the data layer. The data is the data; what changes the dashboard's character is the system prompt.

Chapter 15: The Tools The Agent Has

Pi agents register tools via a TypeScript extension. The wells narrator gets a small, carefully chosen tool kit — scoped to read + operational hygiene, not destructive lifecycle ops. The extension is .pi/extensions/wells-tools.ts.

ToolWhat it does
query_welldGET welld's facts endpoints — the narrator's eyes on the substrate.
well_doctorRun well doctor with appropriate flags — flush stale leases, sweep orphans.
read_sqliteQuery the cell's own SQLite — read history, trends, previous reads.
well_commandGated well CLI passthrough. A permission-gate hook blocks destructive verbs (destroy, stop) — they never run unattended.
⚠️
Warning

Resist the urge to give the agent destroy for "convenience." An agent that can destroy wells unattended is a foot-gun aimed at production. The permission-gate hook on well_command is load-bearing; keep it.

Chapter 16: The TUI Hatch — well talk

Some operators want to talk to the agent in a terminal, not a browser. That's what well talk is for: a one-line CLI command that drops you into the same narrator agent in your own TUI.

Under the hood, well talk SSHes into the wells cell and runs pi (the Pi TUI) in the agent's project directory, continuing the most recent session. The narrator loop's print-mode passes and your interactive TUI both touch Pi's session store — same agent definition, same memory.

🤓
Technical Stuff

Two Pi processes (the loop running in print mode + the interactive TUI you just opened) against one Pi session store is the subtlety to verify during build. Either confirm they share cleanly (Pi's SessionManager.continueRecent handles concurrent access), or give the TUI hatch its own session lineage. The build spec lists this as an open question.

Part VIThe Part of Tens

Ten Things This Dashboard Is NOT

  1. Not Grafana for wells. No time-series, no panels, no PromQL.
  2. Not a request log. Don't show API calls — show what happened.
  3. Not a wall of dials. The page has prose at the top, on purpose.
  4. Not the cells dashboard. Cells gets cells.cells.md.
  5. Not "the welld admin UI." Welld is the data; the dashboard is the read.
  6. Not always-on. The cell hibernates. Wake-on-traffic is the point.
  7. Not a metrics platform. SQLite holds enough history for the agent — not for you to graph.
  8. Not a checkpoint browser. Checkpoints are a CLI/cells workflow, not part of the substrate's story.
  9. Not destructive. The agent's tools can't destroy unattended. Ever.
  10. Not load-bearing for "is wells up." The bare /up page on welld covers that.

Ten Things The Narrator Might Eventually Get Smarter About

  1. Trend recognition. "You've been at 10 GB committed all week."
  2. Anomaly framing. "Three respawns in five minutes is more than usual."
  3. Image age coaching. "Your base image is two weeks old; rebake soon."
  4. Lease hygiene reminders. "I'd flush the stale leases now if it's been a week."
  5. Capacity foresight. "At this rate you'll hit memory in three days."
  6. Bake-time delta. "Seals went from 3.8s → 5.1s last cycle — disk pressure?"
  7. Boundary respect. Quietly point a user at cells.cells.md when they ask agent-y questions.
  8. Self-aware hibernation. "I'm about to nap. Last read in 8 min."
  9. Multi-host (if/when). "Mini #1 is roomy; Mini #2 just hit 80%."
  10. Voice memory. Notice you ask the same thing every Monday and answer it before you ask.

Ten Gotchas To Have Already Thought About

  1. Auth on the public surface. A page where an agent can act needs real auth (not just welld's local bearer). Block before going external.
  2. Cell wake latency. First load after a long hibernation may take ~1–3s. That's still glance-acceptable; warn if it ever creeps up.
  3. Two Pi processes, one session. The narrator loop and the TUI hatch both touch Pi's session files. Verify the sharing model.
  4. Lit-in-React. pi-web-ui is Lit; Next.js is React. Pick one of the three landing options early.
  5. Welld's /dashboard/data is unauthenticated. Don't put secrets in it. (Already true; keep it that way.)
  6. SQLite WAL on a cell. Use WAL mode. Crash-safe between writer (loop) and reader (Next.js).
  7. Don't write welld state from the cell. The narrator can read welld and run well CLI commands; it must never poke at ~/.wells/ directly.
  8. The agent's well_command permission gate. Don't disable it to "make demos easier." Future you will thank present you.
  9. Renaming wells & URLs. If you rename the cell, the URL changes. The cells dashboard at cells.cells.md deserves the same care.
  10. Voice drift. The narrator's character is the dashboard's identity. Check SYSTEM.md like you'd check a public README.
