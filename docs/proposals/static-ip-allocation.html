Static IP allocation — get welld off vmnet's DHCP

Status: Proposal · Date: 2026-05-12 · From: wells team · To: cells team

Follow-up to the W.13 finding (vmnet's bootpd has a concurrent ceiling of 4) and tonight's "doesn't fix" list. Both #1 (DHCP concurrent ceiling) and #3 (intermittent pool refill DHCP timeouts) on that list collapse to a single root cause: we rely on vmnet's bootpd for IP allocation. This proposes bypassing it entirely.

TL;DR

  • Welld today asks vmnet's bootpd to grant DHCP leases for every new well. bootpd's reliability ceiling is ~4 concurrent grants — above that, races cause timeouts.
  • Welld already has the pinned_ip plumbing (Lever 3) for static IPs. Today it's opt-in for explicit operator pins. Flip it to default — welld manages an IP pool, writes static config via cidata netplan, the guest never does DHCP.
  • This makes #1 (concurrent ceiling) and #3 (intermittent refill timeouts) on the "doesn't fix" list obsolete by construction.
  • No cells-side API changes. Cells continues calling well info for IPs; the source becomes welld's pool instead of /var/db/dhcpd_leases.

In plain English

For anyone who doesn't live in this codebase

The problem: when we create a new Linux machine on the Mac, the machine asks the Mac's built-in network service ("bootpd") for an IP address. This is the standard DHCP dance — the machine boots, says "hi, I'm new, what's my IP?", and bootpd gives it one. We've known for a while (W.13 in early May) that the Mac's bootpd can only handle about 4 of these requests at once before it gets confused and some requests just silently fail. Tonight's intermittent "pool refill keeps timing out" failures look like more of the same.

The fix: stop asking bootpd. Pick the IP ourselves before the machine even boots. We already have the code to do this — we built it months ago for a different reason and have been using it for special "pinned" wells. We just never turned it on by default.

Why this works: the Mac's networking lets us assign any IP we want to a virtual machine as long as we tell the machine its IP at boot time (via a tiny config file we already attach to every new machine). We pick an IP from a range we own, write it into the boot config, and the machine starts up with that IP already set. Bootpd is bypassed entirely. The "concurrent ceiling of 4" goes away — we can spin up 50 machines at once without any race condition, because nothing's racing.

What changes for cells: nothing visible. Cells's code asks wells for a cell's IP via the existing API; the answer comes faster and more reliably. No changes to vhost dispatch, talk path, or birth flow. The only thing that changes is that we stop seeing "no DHCP lease for hostname egg-XXX within 90000ms" errors.

What changes for wells: wells maintains a small ledger of which IPs are in use. New wells get the next free IP. Destroyed wells return their IP to the pool. The lease file (/var/db/dhcpd_leases) becomes irrelevant for any well wells creates — bootpd still uses it for any other VMs on the system, but wells doesn't read or write it. Tonight's W.68 incident class goes away by construction.

Effort: roughly a day and a half of wells-side work. Zero cells-side work. The migration is gentle — existing wells keep their DHCP-assigned IPs until they're destroyed; new wells use static. No flag-day cutover.

Mechanism

IP pool

vmnet's default subnet on macOS is 192.168.64.0/24. Bootpd's DHCP range within that is configurable in /Library/Preferences/SystemConfiguration/bootpd.plist — typical default is .2-.254 but operator config can narrow it. Welld carves out a non-overlapping range for its static allocations.

Proposed defaults: welld claims 192.168.64.200-250 (51 IPs). Verifiable at welld startup — read bootpd.plist, abort if our range overlaps bootpd's declared range. Operator-configurable via ~/.wells/defaults.json: "static_ip_range": "192.168.64.200-250".

Allocation

Welld registry already has the pinned_ip field per well. Currently opt-in (set via POST /v1/wells {pinned_ip}). Proposal:

  1. On POST /v1/wells, if request doesn't specify pinned_ip, welld allocates from the pool: walks the registry, finds an unused IP in the static range, stamps it on the new well's record.
  2. Allocation under a per-welld mutex (no race — single welld process).
  3. On DELETE /v1/wells/<name>, the IP is implicitly released (registry record gone = IP available again).
  4. Concurrent creates are serialized at the allocation step only; the rest of handleCreateWell proceeds in parallel. No DHCP race because no DHCP.

Cidata netplan

Current cidata template writes:

network:
  version: 2
  ethernets:
    enp0s1:
      dhcp4: true
      dhcp-identifier: mac

Proposed: when welld has a static IP, write:

network:
  version: 2
  ethernets:
    enp0s1:
      dhcp4: false
      addresses: [192.168.64.215/24]
      routes:
        - to: default
          via: 192.168.64.1
      nameservers:
        addresses: [192.168.64.1]

Gateway + nameserver are vmnet's bridge address (consistent at 192.168.64.1 on Apple's shared-mode vmnet). Static at host level — verifiable from netstat.

Where the IP comes from at read time

Welld's resolveWellIp(name) already prefers pinned_ip over the lease file:

// from lib/dhcp.ts
if (record?.pinned_ip) return record.pinned_ip;
// ... falls through to MAC + hostname lookup in lease file

Cells's vhost dispatch calls this. No change required on either side — once pinned_ip is populated on the registry record, resolveWellIp returns it instantly without touching /var/db/dhcpd_leases.

Today vs proposed

Today (DHCP)

  • welld writes cidata with dhcp4: true
  • lume.start creates VM
  • guest's netd brings up enp0s1, dhclient sends DHCPDISCOVER
  • bootpd responds with offer (or races — N≥5 partial-fails)
  • welld waits up to 90s for lease entry to appear in /var/db/dhcpd_leases
  • SSH-ready check via observed IP

Bootpd in the critical path. Race-prone at N≥5. Intermittent failures even at N=1.

Proposed (static)

  • welld allocates IP from pool, stamps on registry record
  • welld writes cidata with dhcp4: false + static address
  • lume.start creates VM
  • guest's netd brings up enp0s1 directly with the static address
  • SSH-ready check at known IP — no waiting for DHCP

Bootpd bypassed. No race. IP known before guest boots.

Migration story

No flag-day cutover. The transition is gentle:

PopulationTreatment
Wells created before this landsKeep DHCP. Their pinned_ip is unset; resolveWellIp falls through to lease lookup as today. They live out their natural lifetime on DHCP.
Wells created after this lands, no pinned_ip in requestGet auto-allocated static IP from welld's pool. Cidata netplan is the static variant.
Wells with operator-specified pinned_ip in requestSame as today — operator's pin honored exactly. (This path already exists; we're just changing the default.)
Pool members (post-slice-2: owned by cells)Cells's pool creates wells via POST /v1/wells → automatically static-IP. Cells does nothing different.

Mixed environment for some weeks. Both DHCP-allocated and static wells coexist on the same vmnet bridge; macOS handles this natively as long as the IP ranges don't overlap (verified at welld startup).

Open questions for cells team

Q1 — Anything you read from /var/db/dhcpd_leases directly?

If yes, that path breaks for static wells (they won't have entries). If you go through well info or resolveWellIp, you're fine — no change.

Q2 — Pool members: any preference on when they get their IP?

Current proposal: when cells calls POST /v1/wells to bake a pool member, welld picks an IP, returns it in the response. Cells doesn't need to know it. Alternative: cells explicitly requests a pin. Either works.

Q3 — IP range

Proposed: 192.168.64.200-250 (51 wells). If cells expects to exceed that, we expand. Operator-configurable. Concurrent ceiling becomes pool-size-limited instead of bootpd-limited.

Q4 — Migration timing relative to slice 2 of boundary cleanup

Static IPs should ideally land BEFORE cells takes pool ownership (slice 2). Otherwise cells inherits the DHCP-failure problem mid-migration. Suggested order: ship W.72 → cells team's slice 2 work. Workable?

Recommendation

Ship W.72 as soon as cells team confirms Q1-Q4 don't reveal a blocker. Sequence-wise, this should land before slice 2 of boundary cleanup so cells inherits a working pool primitive (not the DHCP-failing one).

If cells team's answers reveal blockers, fall back to W.71's observability as the holding action — diagnose the intermittent failures, work around them. But W.72 is the architectural fix; observability is just a hedge.
