Boundary cleanup — Pieces 2 & 3

Explainer for Pete · 2026-05-13 post-V1 · written from the original proposal at docs/proposals/wells-cells-boundary-cleanup.html

What this is

  • The boundary cleanup is a three-piece plan to untangle wells's code from cells's code. Piece 1 (delete the lease publisher) shipped on 2026-05-12.
  • Piece 2 is the big one: move the "pool of pre-baked wells" out of wells's codebase and into cells's. ~1,148 LoC deleted from wells, ~2-3 days of cells-side work.
  • Piece 3 is a small wells-only follow-on: once cells owns the pool, wells doesn't need its "warming sequence" anymore. ~6-8s faster well create, ~100 LoC simpler.
  • Both pieces are coordinated, not autonomous — wells can't unilaterally ship Piece 2.

In plain English

For the version that doesn't require living in the codebase

Wells builds Linux virtual machines on your Mac. Cells runs AI agents inside them. Today, wells does extra work that's really about cells's needs — most prominently, it maintains a stash of "pre-built apartments" so cells's agents can move in quickly. That stash is called the pool.

The pool living in wells is technically convenient but architecturally wrong. The pool is a cells thing — what's in it, how it's furnished, when to refill — all cells decisions. Wells just happens to own the code because that's where it landed when it was first built.

Piece 2 fixes that. Cells builds its own pool manager, using wells's primitives (create a VM, hibernate a VM, wake a VM) the same way any client would. Wells deletes about 1,100 lines of "pool" code that was always cells's responsibility.

Piece 3 is a small cleanup that becomes possible after Piece 2. Right now wells does some extra boot-shuffle work so every well it creates is "hibernate-ready." That extra work exists only because wells's pool wanted hibernated wells. Once the pool moves out, the boot-shuffle goes too — fresh-create gets ~6-8 seconds faster.

Why it matters: after these land, wells is "the VM platform" and cells is "the agent platform on top." Each team can move faster on its own. The W.68-class bug (where wells reached into something cells-shaped and broke the substrate) becomes impossible by construction — wells can't break cells's rules if it doesn't hold any.

Piece 2 Move the pool to cells ~1,148 LoC delete from wells · ~2-3 cells-days

Plain English

Wells today keeps a stash of pre-baked Linux VMs that have cells's agent stack already installed and are hibernated (RAM saved to disk, ready to wake in ~1-2 seconds). When cells wants to create a new agent, it grabs one from the stash instead of waiting ~10 seconds for a fresh VM. After Piece 2, cells maintains that stash itself. Wells just provides the primitives — "create a VM," "put a VM to sleep," "wake a VM up" — like any infrastructure layer.

What ships

Wells deletes

  • lib/poolFill.ts — 459 LoC
  • lib/poolFiller.ts — 229 LoC
  • lib/poolRegistry.ts — 230 LoC
  • lib/adoptFromPool.ts — 230 LoC
  • ~/.wells/pool/ state dir
  • GET /v1/wells/pool + POST /v1/wells/pool/refill REST routes
  • well pool CLI subcommands
  • 50-80 pool-related tests
  • Maybe: lib/identityReset.ts — 109 LoC (open question Q2 from the proposal)

Cells builds

  • Pool manager in cells-side code (likely the cells CLI/daemon)
  • Pool state at ~/.cells/pool/ (their call — Q1 in proposal)
  • Refill loop: POST /v1/wells + their bake recipe + POST /v1/wells/<n>/hibernate
  • Birth flow: pick a hibernated egg → POST /v1/wells/<n>/wake → SSH in via well exec → rotate identity
  • Pool depth + target heuristics (already cells's domain)

The REST contract — unchanged for clients

Cells already calls POST /v1/wells, /hibernate, /wake, /exec, DELETE /v1/wells/<n>. Piece 2 doesn't add any new endpoints — it just removes the pool-specific ones, which cells consumes today but didn't really need to.

Why it's coordinated

Wells can't delete the pool code until cells has its replacement working. The sequence: cells ships pool manager → wells deletes pool code → both sides re-run V1.5/V1.10 acceptance to confirm nothing regressed. Cells team owns the timing.

Piece 3 Simplify createWell's warming sequence ~100 LoC delete · ~30 min wells-only

Plain English

When wells creates a fresh VM, it currently does a "warm-up dance": boot once with the configuration disk attached, let cloud-init run, shut down, boot again WITHOUT that disk, wait for SSH. That dance exists because Apple's hibernate doesn't survive having the configuration disk attached — so wells made every fresh VM hibernate-compatible by default. The dance adds ~6-8 seconds to every well create.

After Piece 2, the only consumer of "hibernate-ready by default" is cells's pool — and cells's pool will do its own bake (which already detaches the configuration disk). Operator-created wells (Pete typing well create my-test) don't need to hibernate, so they don't need the dance. Drop it. Fresh creates get ~6-8 seconds faster.

What ships

  • Remove the "stop → restart-without-mount → second-DHCP-wait" block from lib/createWell.ts (currently lines ~605-650)
  • Operator-created wells boot with cidata.iso and run as alive_running. Period.
  • Cells's pool is already responsible for detaching cidata in its own bake before calling hibernate (already true in their existing code path).
  • ~100 LoC simpler. Probably 3-5 tests deleted.

Why it waits on Piece 2

The warming sequence is currently called by wells's own pool filler. If we delete the warming sequence before Piece 2 ships, wells's pool would start producing wells that can't hibernate — and break wells's own machinery. Piece 3 has to wait until Piece 2 has moved the pool out.

My take — what I think the next big piece is

  1. Piece 2 is the right call architecturally. The pool living in wells was a Phase A.1.4 expedient — fast to ship, slow to maintain. The W.68 incident on 2026-05-11 demonstrated the cost concretely: wells reached into substrate plumbing to enforce a cells-shaped invariant and took out 4 of 5 cells in the process. That class of bug doesn't exist if wells stays out of cells-shaped state.
  2. But Piece 2 is not wells-team-autonomous. Wells can't ship Piece 2 unilaterally — the deletion has to land AFTER cells has its replacement working. From wells's side, this is half a day of focused deletion plus regression testing. From cells's side, it's ~2-3 days of focused work. Cells team has signaled willingness; timing is theirs to set.
  3. What wells team CAN do now without waiting:
    • Answer the proposal's five open questions (Q1-Q5) so cells has a clear contract to build against.
    • Pre-stage the wells-side deletion on a branch (feature/piece-2-pool-delete) so it's ready when cells gives the go-ahead.
    • Make sure the existing REST endpoints cells needs (create, hibernate, wake, exec) are documented + tested at the level of detail cells will need to build against them.
  4. Piece 3 is a no-brainer that ships right after Piece 2. No reason to plan it separately — it's a mechanical follow-on that should land in the same wells-side commit window.
  5. Don't try to be clever and ship Piece 3 before Piece 2. Wells's own pool filler is currently the consumer that needs the warming sequence. Removing warming first breaks wells. The proposal's ordering exists for a reason.

If you want one sentence: Piece 2 is the substantial work, it's the right architectural move, but execution is gated on cells team bandwidth — wells's job for the next few weeks is to make Piece 2 as easy as possible for cells to consume.

What "done" looks like

After Pieces 2 + 3 ship, the layout is:

wells

  • VM lifecycle: create, start, stop, hibernate, wake, destroy
  • Image store + R2 sync
  • DHCP read-only (observe leases, don't write)
  • Engine boundary (lume / vwell; future Firecracker)
  • REST API + well CLI + WS vhost dispatch
  • Smaller: ~1,200 LoC lighter, 50-80 fewer tests, no pool concept anywhere

cells

  • Pool manager: refill, drain, depth target, variant mix
  • Bake recipe: DNA install, agent stack setup
  • Birth flow: wake + SSH identity rotation
  • Reachability recovery: if their cell goes dark, decide stop+start vs destroy+recreate
  • Their pool, their rules, their iteration speed
