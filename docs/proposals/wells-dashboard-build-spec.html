Wells Dashboard

A calm, story-first view of the wells substrate — a Pi-agent-backed Next.js app, running on a cell, on a well.
In plain English: wells runs Linux boxes on a Mac. This dashboard is how you glance at that Mac and instantly know it's healthy and has room — and, when you want, talk to an agent that watches it and can fix small things. It is deliberately not a wall of graphs. It tells a story, and explains the system while it does.

Two build tracks run in parallel: CELL the wells-narrator cell builds the app + agent  ·  WELLD the wells team extends welld's facts API. Each section is tagged.

1The concept

The dashboard does two jobs at once, every glance:

  • Tells you how things are going — operational status, the Mac's memory, what changed.
  • Explains the system — the structure is the frame; every glance re-teaches what wells is.

What it is not:

  • Not an IT dashboard. No gauges, dials, metrics walls, raw log tails, time-series charts. The existing lib/dashboard.ts renderDashboardHtml() is exactly the thing being replaced.
  • Not a request log. Never "POST /v1/wells was called." Show what happened — "a well was born."
  • Glance-first. The story page is a five-second read. Depth is one click away, never in your face.
The dashboard's job is comprehension, not monitoring.

2The boundary — what belongs on it

This is wells's view: the substrate. Not cells's. The line is the wells API — the membrane.

  • In scope: everything that crossed the wells API + all wells-side and host state — well lifecycle, the host's resources, substrate health, the image library, what changed.
  • Out of scope: cells's reasoning, cells-to-cells coordination, what an agent is doing inside a well. Wells sees vital signs; cells has the patient's chart.
  • Report the membrane, not the motive. "Three wells born this hour" is a wells fact. "Cells scaled up because traffic spiked" is a cells fact — don't guess it.
  • In-scope ≠ worth-showing. The API defines what's eligible; the agent filters for what means something. State changes matter; exec volume is wallpaper until it fails.
  • No caller attribution. welld uses one shared bearer token; it can't tell cells from the operator. Don't build "cells did X vs you did X."

3Architecture

The dashboard is a Next.js app inside a cell — the wells-narrator cell — backed by self-hosted Convex on that same cell. welld stays a dumb, fast substrate daemon that just exposes facts.

Pivot from the first draft: the original plan said SQLite; we swapped to Convex (Pete's call, 2026-05-14) so every mutation propagates live to every browser tab via useQuery — no polling, no refresh. Same on-cell architecture; data layer just learned to push.

Degradation — load-bearing

If the dashboard is wholly a cell and the substrate is down, the dashboard is down with it — it can't report its own foundation collapsing. So welld keeps a bare, dumb, always-on truth-path: /healthz (public, JSON) plus /up (public, HTML — landed in commit 01c39fa). On the rich pages, if the narrator is asleep or unreachable, "the read" goes quiet gracefully; everything else still renders from the latest Convex snapshot.

4The wells-narrator agent CELL

wells-narrator is a Pi agent (@mariozechner/pi-coding-agent) — Pi is the harness. It runs in two modes against one agent definition:

ModeHowFor
The narrator passPeriodic. The loop invokes Pi in print mode (pi -p) or via the SDK (createAgentSession) — one interpretation pass, capture output, done.Writing "the read" and "lately" — passive narration.
The talk sessionLong-lived. pi --mode rpc as a child process the Next.js app drives over JSONL — or the SDK held by a Next.js server route.The talk page — interactive, and the agent's tools are live so it can act.

On Pi and the web: Pi doesn't ship a hosted chat product — but it does ship a first-party web-UI component library, @mariozechner/pi-web-ui (a ChatPanel component + storage backends + tool/dialog rendering, versioned in lockstep with the CLI). The talk page builds on that — see §9. (The base image's pi-web-access is unrelated: a web-browsing tool for the agent, not a UI.)

The agent definition (.pi/)

  • AGENTS.md — project context: what wells is, the boundary rules, what the narrator's job is. Pi concatenates this into the system prompt.
  • SYSTEM.md (or APPEND_SYSTEM.md) — the narrator's voice and rules: operational-first, plain English, flag warts calmly, never show raw API calls, the wells/cells boundary. This is where the "not an IT dashboard" discipline is enforced.
  • .pi/settings.json — provider + model (a small fast model is fine for narration; the talk session may want a stronger one).
  • .pi/extensions/wells-tools.ts — the agent's tools (below).

Tools — the wells-tools.ts extension

A Pi extension registering the tools the agent uses. Scope them to read + operational hygiene — not destructive lifecycle ops, at least to start.

ToolDoes
query_welldGET welld's facts endpoints — the agent's eyes on the substrate.
well_doctorRun well doctor / flush stale + orphan leases.
read_sqliteQuery the dashboard's own SQLite — history, trends, past reads.
well_commandGated well CLI passthrough. A tool_call hook (Pi's permission-gate pattern) blocks destructive verbs — destroy, stop — requiring they never run unattended.

Hibernation — the agent costs nothing idle

Between narrator-loop ticks, the cell can hibernate. It wakes on the timer (or a welld event), runs a pass, sleeps. The dashboard's intelligence runs on wells's own value prop — and the narrator shows up in its own portrait as a sleeping well, which is honest and teaches.

Auth

Pi needs an LLM key — seed ANTHROPIC_API_KEY (or write ~/.pi/agent/auth.json, 0600) into the cell at create time via wells's --env support. Set PI_TELEMETRY=0.

6Data contracts

welld facts API — available today WELLD

GET /dashboard/data — public, no auth. Current shape (see lib/dashboard.ts DashboardData):

{
  generated_at: string,
  daemon: {
    version, started_at, uptime_seconds, degraded,
    lume: { base_url, owned, respawns_last_hour, respawns_last_5min, respawns_last_1min },
    vz_xpc_count: number
  },
  wells: [{ name, status: "running"|"stopped"|"missing", ip, created_at, last_running_at }],
  vmnet_leases: { total, orphan_count, orphans: [{ name, ip }] },
  images: [{ name, aliases, size_bytes, created_at, from_well, rinsed, firstboot_supports_static_ip }],
  events: string[]   // raw welld.log tail — agent must lift to "lately", never show raw
}

GET /v1/wells/{name} — bearer auth — adds per-well: uuid, url, cpu, memory, disk_size, disk_used_bytes, auto_sleep_seconds.

welld facts API — to be added WELLD gaps

SQLite schema CELL

The cell owns this. Starting shape — adjust as the build reveals needs:

snapshots     -- one row per welld poll: the timeline
  id, polled_at, raw_json,
  well_count, running_count, stopped_count, hibernating_count,
  committed_mb, host_total_mb, degraded

reads         -- one row per narrator pass: the read history
  id, written_at, body, based_on_snapshot_id

events        -- the "lately" timeline, narrated
  id, occurred_at, kind, summary, well_name?, snapshot_id

wells         -- current known wells (upserted each poll)
  name, status, runtime_state, ip, cpu, memory_mb, disk_size_gb,
  disk_used_bytes, memory_resident_bytes, uuid, url,
  auto_sleep_seconds, created_at, first_seen, last_seen

well_timeline -- per-well lifecycle events for the detail page
  id, well_name, occurred_at, kind, summary

images        -- the image library, snapshotted
  name, aliases, size_bytes, origin, created_at, r2_status, wells_forged

metrics       -- timed operations (once welld exposes timing)
  id, op, well_name, duration_ms, occurred_at

7The narrator loop CELL

loop/narrator.ts — a long-lived process. Each tick:

  1. Poll welld — GET /dashboard/data, plus GET /v1/wells/{name} for any well missing sizing. Write a snapshots row.
  2. Diff against the previous snapshot — new wells, state changes, lease changes, image changes, degraded flips, the welld events tail.
  3. Decide — run a Pi pass if something meaningful changed or it's been ≥ N minutes since the last read (start: ~3–5 min quiet cadence, immediate on change).
  4. Pi pass — invoke Pi (print mode or SDK) with: the current snapshot, the diff, the last read. The agent returns the new read (prose) and any new lately events (structured). Its SYSTEM.md carries the voice + rules.
  5. Write SQLite — reads, events, upsert wells / well_timeline / images / metrics.
  6. Sleep — until the next tick or a wake signal. The cell may hibernate in between.
In plain English: the loop is the heartbeat. It checks welld every few minutes, and when something's actually different, it asks the Pi agent "what's the story now?" and saves the answer. The website just reads those saved answers — so the website is always fast, and the thinking happens on its own schedule.

8.2 · images

One noun: images. One list. The base image is row one, flagged by its role ("12 wells forged from this") — not a different category. Saved images are more rows, differing only by origin. R2 is a column (is it backed up?), not a category. No "blueprint," no "saved snapshot." Data: the images table.

9The talk surface CELL

The only page with live machinery — and it builds on Pi's first-party web-UI component library, not a hand-rolled chat. @mariozechner/pi-web-ui ships a ChatPanel component (plus storage backends and tool/dialog rendering). It already handles the conversation, streaming, and — critically — tool executions as action blocks, which is exactly the chatbot-vs-agent line we want: the transcript shows the agent did something and the result.

Recommended — pi-web-ui + the SDK

The maintainer's own pattern: instantiate the agent in-process via pi-agent-core (the SDK), hand it to <ChatPanel>. Start from the working example at pi-mono/packages/web-ui/example — copy it, point it at the wells-narrator agent definition (same .pi/, same wells-tools.ts as the loop). One agent, two surfaces.

Known consideration — Lit inside a React app

pi-web-ui is built on Lit (web components); the dashboard is Next.js (React). Three ways to land it: (a) use <ChatPanel> as a custom element inside the React app — web-component interop, works but a seam; (b) make the talk page a small standalone Lit/Vite app the way the upstream example is, linked from the nav; (c) skip pi-web-ui entirely and drive pi --mode rpc behind a Next.js API route, building the chat UI natively in React. (a) and (b) reuse Pi's maintained rendering; (c) is full control at more cost. Decide during build — (b) is closest to the upstream example and the cleanest start.

If you go RPC (option c)

Spawn pi --mode rpc as a child process (with wells-tools.ts loaded); web/app/api/talk/route.ts bridges the browser to it — user message → RPC prompt → stream message_update / tool_execution_* back. RPC framing is strict \n-delimited JSONL: do not use Node's readline (it splits on U+2028/2029 and breaks the protocol). Roll a splitter.

Note: pi-web-ui has no published npm README — its docs are the in-repo README + the example. Expect to read source.

10The TUI hatch CELL

A copy-able command — well talk — that drops you into the same agent in your terminal. Mechanism: it SSHes into the wells-narrator cell and runs pi (the TUI) in the agent's project dir, continuing its session (SessionManager.continueRecent). The loop's narration passes and the interactive TUI both work against Pi's session store — note this as a known subtlety to verify (two Pi processes, shared session files). well talk is a small new wells CLI verb.

11welld changes WELLD

  • Delete renderDashboardHtml() from lib/dashboard.ts — welld stops serving a dashboard.
  • Keep + extend buildDashboardData() / /dashboard/data — add the four gaps from §6: host memory, per-well resident memory, runtime_state, op timings.
  • Add a bare /up page — minimal HTML, served by welld directly, no agent: "substrate up · N wells · welld vX · up Yh." The always-on degradation path.
  • Leave /healthz and /v1/wells as they are — already the right shape.

12Visual language locked

Warm, calm, structure over graphics. The full token set + components are in dashboard-mock/style.css — reuse it, port it into the Next.js app, don't reinvent. Semantic tokens: teal = the wells system · amber = awake / holding memory · cool slate = stopped / idle · terracotta = needs a glance (never alarm-red) · green = operational.

Real warts show through. The mockups carry real data including the messy parts (178 orphan leases, doubled IPs). The narrator contextualizes them — "worth a glance, not urgent" — rather than hiding or alarming. That's the AI layer earning its keep.

14Open questions decide before / during build

  • Who births and runs the dashboard cell? Wells can do it directly (create + exec + images), or cells manages it. The dashboard is a wells artifact; running a Next.js app on a cell is cells's wheelhouse.
  • Auth on the public surface. An agent that can act, reachable at a public URL, needs real auth on that surface — separate from welld's local bearer. Required before the talk page goes external.
  • The URL. wells.dashboard.cells.md is the working name — confirm.
  • Model choice. Narrator passes can use a small fast model; the talk session may want a stronger one. Two models, one agent definition.
  • Two Pi processes, one session store. The loop and the TUI hatch both touch Pi's sessions — verify the sharing model or give the TUI its own session.
  • Lit inside React (§9). Whether the talk page uses pi-web-ui as a web component in the Next.js app, runs as a standalone Lit app, or goes native-React over RPC — a build-time call.

15Milestones

Phase 1 — the story, end to end

WELLD extend /dashboard/data with host memory + runtime_state. CELL scaffold the Pi project + Next.js + SQLite; build the narrator loop; build the story page.

Done when: the story page renders real data from SQLite, and "the read" updates on the loop's timer with the narrator's actual prose.

Phase 2 — images + well detail

CELL the images list and the /well/[name] drill-down, both from SQLite. Nav wired.

Done when: both pages render from SQLite; clicking a well on the story page opens its detail.

Phase 3 — talk

CELL the wells-tools.ts extension; the talk page built on pi-web-ui's ChatPanel + the SDK (start from packages/web-ui/example) — or RPC if that's the chosen seam. Action blocks come for free with pi-web-ui.

Done when: you can ask the agent a question and get an answer, and tell it "flush the stale leases" and watch it run well doctor and report back.

Phase 4 — ship

CELL the well talk TUI hatch. WELLD the /up degradation page + op timings. Both: the public URL + auth on the talk surface.

Done when: well talk opens the agent in a terminal; the dashboard is reachable at its URL with auth; and with the narrator cell stopped, the story page still renders the well-house and lately from the last snapshot — only "the read" goes quiet.
