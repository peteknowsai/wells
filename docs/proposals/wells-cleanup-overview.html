1What we provide

The Linux substrate that cells lives on

Wells is the platform layer underneath cells. We build the empty Ubuntu Linux apartments, manage their lifecycles, and expose them to cells through a clean API. Everything cells does — the AI agent, the pool, the birth flow — happens inside apartments we maintain.

From wells's perspective, cells birth looks like this: a REST call lands at our daemon, we hand cells a hibernated VM, cells wakes it and installs their identity, the cell answers traffic. Our job is to make those primitives reliable, fast, and predictable. We don't know what the agent inside is doing; we just make sure the box runs.

Two layers, one API between them. Wells doesn't care what cells installs above; cells doesn't care how wells runs the VM below.

2Who builds what

Two teams, two layers

The clean way to think about it is property manager and moving company. We're the property manager: build the apartments, plug in the plumbing, keep them running. Cells is the moving company: furnish them, change the locks, move tenants in.

The platform team — us

Wells

Property manager — builds & rents apartments
  • Builds and maintains bare Ubuntu Linux apartments (the VMs)
  • Starts, stops, hibernates, wakes them on demand
  • Manages disk images (ubuntu-base + image storage for cells's images)
  • Owns the networking layer — IP allocation, lease state, DNS
  • Exposes a clean REST API; doesn't peek inside the apartments
The product team

Cells

Moving company — furnishes & moves tenants in
  • Decides what's installed inside each apartment (the AI stack)
  • Pre-furnishes a few apartments and keeps them ready (the pool)
  • Picks one and changes the locks on cells birth
  • Talks to the agent on the user's behalf
  • Recovers if an agent becomes unreachable

The empty apartment is wells's. Everything in it is cells's. That's the rule.

3What we're cleaning up

We blurred the boundary, then it bit us

For convenience, wells has been doing some moving-company work — specifically, maintaining the pool of pre-furnished apartments. We put that code on our side because welld was the only daemon at the time. Yesterday it cost us.

Yesterday's incident · 2026-05-11 09:01 UTC

We shipped a change called the lease publisher that was supposed to enforce a rule about agent reachability — "if welld says a VM is running, its IP should work." Within minutes, the publisher was poking macOS's networking subsystem about 96 times per minute, and that traffic crashed in-progress address handshakes for cells's other VMs. Four of five running agents went dark.

We mitigated in five minutes (disabled the publisher), shipped a proper batched-kick fix shortly after, and one VM that was being baked during the storm needed to be destroyed and re-baked. The deeper question — why was wells trying to hold a rule that's really cells's concern? — is what this cleanup answers.

What we're keeping clear about: wells still owns the networking layer. We're not stepping back from it — we're actually owning it more directly (see section 5: we're going to allocate IP addresses ourselves instead of asking macOS to do it). What's moving to cells is the cells-shaped question — "is my agent reachable, and if not what do I do about it?" — not the underlying plumbing.

The same pattern shows up at a bigger scale with the pool of pre-furnished apartments. We maintain the pool today, but the contents are cells-shaped (which agent stack, which models, which identity). The cleanup moves the pool the same way: wells keeps the apartments themselves, cells decides what's in them and who gets which one.

4The cleanup, in three pieces

Smallest first, then the main piece, then a small follow-on

Piece 1 · wells

We delete the lease publisher

~30 min · wells only

Remove the code from yesterday's incident permanently. Already disabled in production; this is the deletion. No dependencies, ships anytime this week.

Cells work: none.

Piece 2 — the big one

Cells takes over the pool

~2.5 days · cells

The pool of pre-furnished apartments moves out of wells and into cells. About 1,100 lines of code shift over; most of the machinery is already on cells's side, so this is consolidation rather than a rewrite.

After this piece, wells is purely a VM platform. Cells is purely the AI agent product on top.

Wells work: delete our copy after their PR merges (~half day).

Piece 3 · wells

We simplify VM creation

~30 min · wells

Once the pool is cells's, we can simplify how we create fresh VMs — we no longer have to make every VM hibernate-ready by default. Slightly faster fresh births as a bonus.

Cells work: one regression test, ~1 hour.

5What wells is shipping alongside

Two substrate improvements that come with the cleanup

Bonus 1 — independent of the cleanup

Static IP allocation

Planning the cleanup, we looked at the substrate and found a long-standing limit: macOS's networking can only assign IPs to four VMs at a time before races cause timeouts. Source of the intermittent failures we've all seen when refilling the pool fast.

We're fixing it by allocating IPs ourselves instead of asking macOS. Welld picks from a managed pool (192.168.64.200–250 by default), writes static config into each VM's boot disk, bootpd is bypassed entirely. About 11 hours of wells-side work. Cells doesn't change anything — same REST calls, same well info. Lands before Piece 2 so cells inherits a working substrate from day one.

Bonus 2 — visibility

Operator dashboard

A live HTML page at localhost:7878/dashboard that polls the substrate and renders state visually — daemon health, wells grid with status per cell, pool depth, DHCP/IP state, recent events. Open in a browser; auto-refreshes every few seconds.

Replaces the current curl-and-grep workflow with a single visual surface. About 3–4 hours of wells-side work. Ships early this week so it's live for the static-IP deploy.

7What could go sideways

Three categories, all bounded

Already mitigated
Yesterday's lease-publisher class of bug

Disabled in production at 09:01Z, batched-kick fix shipped by 09:04Z, code being deleted permanently in Piece 1. Substrate stable on the current stable rev.

Wells eliminating
The 4-VM networking ceiling

Static IPs ship before Piece 2 lands. Cells inherits a substrate without the ceiling — so concurrent fan-out becomes pool-size-limited (50+ IPs available), not bootpd-limited (4).

Watching
Identity-rotation port

Our SSH-side identity reset code moves to cells in Piece 2. They'll port it verbatim; both teams will verify that two consecutive births produce unique identities before wells deletes our copy. Edge cases (forks-from-saved-image, machine-id rinse) are well-covered by tests but worth one extra round of attention during the cutover.

If you've heard these terms in our codebase

W.71
The diagnostic-dump feature wells shipped earlier today. When pool refill fails, welld now logs a full snapshot (lume state, lease state, bundle MAC, disk hold) so root-causing intermittent failures doesn't require re-reproducing them.
W.72
Wells's static-IP work described in Section 5 of this overview. The "bonus" fix that makes the 4-VM networking ceiling go away.
Piece 1 / Piece 2 / Piece 3
The three pieces of the cleanup, in order: delete the publisher (wells), migrate the pool to cells (cells's big lift), simplify VM creation (wells, small). Same as cells's "Slice 1/2/3" — different word, same thing.
The lease publisher
The wells-side component that caused yesterday's incident. It tried to keep cells's running VMs reachable by frequently editing macOS's network lease file. The editing turned out to crash bootpd's in-flight handshakes. Deleted permanently in Piece 1.
The pool
The small set of pre-furnished apartments (pre-warmed VMs with cells's agent stack installed) that cells birth hands off in three seconds. Currently managed by wells; moving to cells in Piece 2.
ubuntu-base / cell-base / variant egg
Three layers of disk image. ubuntu-base is wells's bare Ubuntu (the empty apartment); we publish a mutable alias for stable consumption + immutable versioned tags for reproducibility. cell-base is cells's: Ubuntu plus the agent stack (the furnished apartment). Variant eggs are cell-base plus model-specific extras.
Static IPs / DHCP
Today: macOS's bootpd assigns IPs to new VMs via DHCP, which races above ~4 concurrent grants. After W.72: welld picks IPs from a managed pool before the VM boots, bypassing bootpd entirely. No ceiling.
Substrate
Wells's layer — Linux VMs running on macOS. "The substrate" = the foundation cells builds on. We use the term to remind ourselves where the boundary is.
Stable rev / wells-stable-YYYY-MM-DDX
The git tag cells pins to when they want a known-good substrate version. Today's is wells-stable-2026-05-11d. We bump the letter when we ship; cells decides when to follow.
