The Cheat Sheet

Everything on one tear-out card. Read this if you read nothing else.

The problem in one sentence
Wells runs create / start / wake with unbounded concurrency, so when too many VM boots land at once the SSH-ready gate times out — which is exactly what paused the cells pool re-bake on 2026-05-14.
What's actually constrained
Not the fleet size — the boot burst. CPU oversubscription just makes VMs slower; it doesn't fail them. Boot is a short CPU-heavy spike with a deadline, and too many spikes at once blew the deadline. A booted, mostly-waiting agent costs almost nothing — so steady-state fleet size is bounded by memory, and 48 GB is roomy (~50–70 wells).
The fix in one sentence
A bounded-concurrency admission gate on the VM-booting primitives: N boots run at once, the rest wait for a slot. The REST caller doesn't change — the call just takes a little longer instead of failing.
It is NOT
A persistent job queue. No disk state, no priorities, no retries, no serialization. It's a semaphore, not a scheduler.
Who owns it
Wells. Wells is the only layer that can see the whole host's load. Cells owns what to bake and when; wells owns how fast the box can physically absorb it.
Smallest shippable version
A static semaphore — WELL_MAX_CONCURRENT_BOOTS, default 3 — that createWell acquires around the lume.start → wait-for-SSH stretch. Roughly a day of work. Even this dumb version kills the failure mode.
The tell that we need it
The hand-written "destroys first" guidance we keep giving cells is a workaround for a missing primitive. Admission control makes that guidance unnecessary.
Defaults and tuning knobs
Two env vars on welld. WELL_MAX_CONCURRENT_BOOTS — hard count of boots in flight (default 3). WELL_BOOT_VCPU_RATIO — committed-vCPU backstop (default 2); when Σ(running wells .cpu) / hostCores ≥ this, the effective limit collapses to 1 until load drops. No code change to tune; bounce welld to pick up new values. Live readout: boot_gate {inFlight, waiting, limit} on /healthz.
Reading the limit like an operator
limit: 1 is not "blocked." It means the vCPU backstop engaged and boots happen serially instead of three-at-a-time — every well create still succeeds, the fleet keeps growing, and the limit recovers the moment a well stops, hibernates, or is destroyed (committed vCPU drops, backstop disengages). Memory is the real fleet ceiling, not the gate.

Introduction

About This Book

On 2026-05-14 the cells team kicked off a pool re-bake — eight wells destroyed, ten fresh ones created. Several of the creates timed out. The cells team, reasonably, hit the brakes and asked wells what was going on.

The short answer: nothing was broken. The box was just doing far more than it had the cores for, and wells had no way to say "hang on, let me catch up." This book is the proposal for giving wells that ability. It is written in plain language first, with the deep technical detail clearly marked so you can skip it or dive in — your call.

🧵
Remember

Status: implemented. Pete approved building it; Phases 1–4 shipped to main (63a3b94). It goes live on the next welld bounce. The two open questions in Chapter 10 still want the cells team's input — they're refinements, not blockers.

Foolish Assumptions

This book assumes that you:

  • Know that a well is a stateful Linux VM, and welld is the daemon that herds them.
  • Know that cells is the agent fleet that runs on top of wells, and that cells owns the warm pool while wells owns the raw VM lifecycle.
  • Accept that a Mac has a fixed number of CPU cores, and that wishing very hard does not create more of them.
  • Would rather a slow, reliable well create than a fast one that throws a TimeoutError five minutes later.

Part IThe Problem (Or, Why Your Bake Timed Out)

Chapter 1: What Went Wrong on May 14th

In This Chapter
  • The bake that timed out
  • Why it was load, not a bug
  • The orphan VZ process — same incident, different symptom

When the cells team ran their re-bake, the host already had 13 live wells on it. Each well is configured with 4 virtual CPUs. That's roughly 52 vCPUs scheduled onto a box with somewhere around 10–12 physical cores — a 4–5× CPU oversubscription before the re-bake added ten more VM boots and a stream of well exec traffic on top.

Under that load, a freshly-created VM boots slowly. Cloud-init crawls. sshd takes its time coming up. And wells's "is SSH ready yet?" gate — patient, but not infinitely patient — fired before the guest was actually reachable. The create threw a TimeoutError.

🧵
Remember

The box never ran out of CPU — CPU oversubscription just makes everything slower, it doesn't fail anything. What failed was a timeout: boot is a CPU-heavy spike with a deadline, and a contended boot crossed wells's SSH-ready deadline before the guest finished. The VMs were booting fine — just slowly.

Sidebar: The numbers, on the box this was filed against

The host is an M5 Pro — 18 cores, 48 GB. With 10 wells live, measured load average sits around 2.8: the box is roughly 85% idle. Each well is configured for 1 GB but actually touches about 440 MB resident while its agent is mostly waiting on the model API. So "52 vCPU on a dozen cores" overstates the squeeze — that 52 is a configured ceiling, not consumed CPU. Idle guests timeshare for free. The box only feels it during a pile-up of boots.

Sidebar: The orphan that gave it away

The same timed-out create also left a VM — and its VirtualMachine.xpc process — running with no registry record. well doctor flagged it and (unhelpfully) blamed "a crashed lume serve," even though lume hadn't crashed once. That was a second, smaller bug: a create that fails after lume.start used to leak. It's since been fixed (commit f4e6daf) — a failed create now reaps its own VM. But the leak was a symptom. The disease is that we admitted more boots than the box could finish in time.

Chapter 2: Why "Destroys First" Is a Smell, Not a Fix

In This Chapter
  • The workaround we keep handing cells
  • Why a workaround that needs a human is a missing primitive

When the cells team asked how to safely re-bake, wells's advice was: do the eight destroys first. Destroying eight wells drops the live count from 13 to 5, which frees enough CPU headroom for the ten creates to finish inside the SSH-ready window.

That advice is correct. It also should not have to exist. "Run your operations in a specific order or the substrate falls over" is the substrate leaking its own scheduling problem onto its caller.

⚠️
Warning

Every time wells hand-delivers ordering guidance to cells, that's a primitive we haven't built yet. Today it's "destroys first." Tomorrow it's a subtly different rule for a subtly different load mix, delivered over a chat channel, remembered by nobody. That doesn't scale.

💡
Tip

Good test for "should the substrate own this?": if the safe way to use a primitive is a sentence you keep repeating to the caller, the substrate should enforce that sentence instead of reciting it.

🧵
Remember

Two layers, not one. The primitive (admission control) is the floor you can't fall through — it makes oversubscription self-pacing. The skill — the wells Claude Code skill, the "how to drive the CLI well" doc — is where load-and-ordering practice lives, so guidance like "destroy before a big re-bake" is written down once, not re-typed into a chat channel each time. The primitive stops you breaking the box; the skill teaches you to use it well even where the primitive doesn't reach.

Part IIThe Fix (Admission Control, Not a Queue)

Chapter 3: Meet the Semaphore

In This Chapter
  • What admission control actually is
  • Why it's a semaphore and not a job queue
  • What the caller sees (almost nothing)
  • Burst control, not population control

Admission control is a gate in front of the expensive, contention-sensitive work. Wells decides how many VM boots can be in flight at once. If a create arrives and all the slots are full, it doesn't fail and it doesn't get rejected — it simply waits its turn, then runs normally.

That's it. It's a counting semaphore: a small number of tokens, you take one before you boot a VM, you give it back when the boot finishes. No tokens left? You wait.

🧵
Remember

The REST API does not change. POST /v1/wells behaves exactly as before — it just might take a bit longer to return when the box is busy. Cells doesn't write a single new line of code to benefit from this.

Why "not a queue"? Because the word queue drags in a whole job-system mindset — persistence, priorities, retry policies, dead-letter handling, a scheduler. We need none of that. We need "don't start the 4th boot until one of the first 3 finishes." That's a semaphore. Reaching for a job queue here would be solving a problem we don't have and importing complexity we'd regret.

Burst control, not population control

Here's the sharpened version of the whole idea: admission control gates the boot burst, and nothing else. It is not a cap on how many wells can exist. A booted well whose agent is waiting on the model API costs almost nothing — that's why a host running ten of them can sit 85% idle. What a host can't absorb is a dozen simultaneous boots, each a short, deadline-bearing CPU spike. Gate the spikes, and the steady-state population is bounded only by memory.

💡
Tip

On 48 GB, "bounded by memory" is generous — at the measured ~440 MB per mostly-idle well, that's room for roughly 50–70 of them. We've been running 10–13. Admission control isn't there to keep the fleet small; it's there so the fleet can grow without boots tripping over each other.

🧵
Remember

If /healthz shows boot_gate.limit: 1, that is not "blocked." It means the vCPU backstop engaged and boots are happening serially — one at a time — instead of three at a time. Every well create still succeeds; the fleet still grows. The limit recovers automatically the moment a running well stops, hibernates, or is destroyed (committed vCPU drops below the ratio, backstop disengages). The only ceiling that actually stops the fleet growing is memory, and at ~440 MB per idle well on 48 GB you have many wells of headroom.

💡
Tip

If serial pacing ever feels too conservative in practice — say, the configured-vCPU accounting overstates real load (it usually does) — bump WELL_BOOT_VCPU_RATIO from 2 to 3, or raise WELL_MAX_CONCURRENT_BOOTS from 3 to 5. Both are env vars on welld; bounce to pick them up. The boot count is the primary signal anyway; the vCPU backstop is there for the genuinely-saturated case.

Chapter 4: The Load-Aware Limit

In This Chapter
  • Why a fixed number is the wrong long-term answer
  • The committed-vCPU heuristic
  • Start dumb, get smart

How many concurrent boots is "safe"? It depends — on the host's core count, on how many wells are already live, and on how much vCPU each of them is holding. A single hard-coded number can't be right for every state of the box.

So the limit should be load-aware: before admitting a boot, wells adds up the vCPU already committed to running wells and compares it to the host's physical core count. If admitting one more boot would push that ratio past a threshold (say, 1.5–2× cores), the boot waits until something frees up.

💡
Tip

Ship the dumb version first. A flat WELL_MAX_CONCURRENT_BOOTS=3 is crude, but it would have completely prevented the May 14 timeout. Get that in, prove it out, then make it load-aware. Don't let the perfect heuristic block the good-enough semaphore.

🤓
Technical Stuff

Concretely, two signals. Primary: a hard count of boots in flight — WELL_MAX_CONCURRENT_BOOTS. Secondary: a committed-vCPU backstop — committedVcpu = Σ(running wells .cpu) vs hostCores = os.cpus().length, admit when (committedVcpu + candidate.cpu) / hostCores ≤ RATIO.

The boot-count is the signal that really matters, because the vCPU backstop is deliberately conservative: an idle live well's configured vCPU isn't actually consumed (Chapter 3 — idle guests timeshare for free), so summing it overstates load. Treat the vCPU ratio as a ceiling that rarely binds, not the primary gate. Creates are mostly I/O-wait (clonefile, ssh-poll) so you genuinely want some concurrency — this is a bound, not a serialization.

Chapter 5: Where It Lives in the Code

In This Chapter
  • One small module, three call sites
  • Engine-agnostic by construction

The semaphore is a small new module in lib/ — call it lib/admission.ts. It exposes "acquire a boot slot" and "release it," and knows how to compute the load-aware limit. It does not know or care whether the engine underneath is lume, Firecracker, or anything else — it reasons about host load, which is engine-independent.

The three places that acquire a slot:

Call siteWhy it boots a VM
createWellFresh VM from a base image — the headline case.
startWellCold well coming back to life — same boot cost.
wakeWell / resurrect passHibernated wells waking, and the welld-startup resurrection burst — a dozen boots back-to-back is precisely an admission problem.
🤓
Technical Stuff

Each call site acquires just around the contention-sensitive stretch — the lume.start → waitForSshReady span — and releases in a finally. It does not hold the slot during cheap setup (validating names, writing seeds) or after the guest is up. The handlers in lib/handlers/ stay pure; the semaphore is just another injected dependency, consistent with the existing deps-injection pattern.

⚠️
Warning

Mind the caller's timeout budget. If cells's HTTP client gives POST /v1/wells only so many seconds, "wait for a slot" plus "boot the VM" has to fit inside it. Either keep the admission wait bounded, or have the create endpoint surface "you're queued" instead of holding a silent connection. This is the one genuine open design question — see Part V, Chapter 10.

Part IIIBoundaries and Politics

Chapter 6: Why This Is Wells's Job, Not Cells's

In This Chapter
  • The whole-host-visibility argument
  • How it composes with the cells pool instead of fighting it

The obvious objection: "cells owns the pool — shouldn't cells throttle its own refills?" No. Here's why.

The load that broke the bake wasn't only cells's ten creates. It was ten creates plus 13 live wells plus a stream of well exec traffic from a wells operator. Cells can pace the work cells generates, but it can't see — and shouldn't have to model — load coming from everywhere else on the box. Wells is the only layer with the whole-host view.

🧵
Remember

This is the same principle as W.72, where wells took over IP allocation instead of letting callers race for addresses. "Don't let callers oversubscribe a shared resource" is a substrate responsibility. CPU is a shared resource. Same shape.

And it composes with the cells pool — it doesn't compete with it. The two systems work on different axes:

Cells's pool logic decides…Wells's admission control decides…
What to bake and when the pool wants membersHow fast the host can physically absorb those boots
Refill cadence, eviction, target pool sizeConcurrency bound based on live host load

Cells asks for ten. Wells runs them as fast as the box allows instead of as fast as the event loop allows. Nobody's logic has to know about the other's.

Chapter 7: What Admission Control Is NOT

In This Chapter
  • Scope fences, so this stays small
  • Not a persistent job queue. No on-disk state. A welld bounce resets the semaphore; in-flight boots already hold real OS resources and finish on their own.
  • Not priorities or fairness. First-come-first-served is fine. We are not building a scheduler.
  • Not retries. A boot that fails for a real reason still fails. The caller decides what to do — admission control only governs when a boot is allowed to start.
  • Not serialization. The whole point is bounded concurrency. One-at-a-time would waste the box.
  • Not a replacement for the cells pool cadence. See Chapter 6 — different axis entirely.
⚠️
Warning

If a review of this proposal starts adding "and it should also do priorities, and persistence, and…" — stop. That's a different, much larger project. This proposal succeeds precisely because it's a semaphore and nothing more.

Part IVRolling It Out Without Drama

Chapter 8: The Phased Plan

In This Chapter
  • Four phases, smallest-useful-thing first
  • What to coordinate with cells, and when
  1. Phase 1 — The dumb semaphore. Add lib/admission.ts with a flat WELL_MAX_CONCURRENT_BOOTS (default 3). Wire it into createWell first. Colocated tests. This alone closes the May 14 failure mode. ~1 day.
  2. Phase 2 — Cover the other boot paths. Extend the gate to startWell, wakeWell, and the welld-startup resurrection pass. The resurrection burst is its own mini-stress-test, so this phase doubles as a real-world check.
  3. Phase 3 — Make the limit load-aware. Swap the flat number for the committed-vCPU-vs-cores heuristic from Chapter 4. Keep the flat number as a hard ceiling override.
  4. Phase 4 — Observability. Surface admission depth in well doctor and /healthz so an operator (and cells) can see "wells is pacing me right now, that's expected" instead of guessing. This is the phase to coordinate with cells — it's the first one they can observe.
💡
Tip

Phases 1–3 are invisible to cells by design — the API contract doesn't move, calls just pace themselves. Only Phase 4 adds something cells can see, so that's the only phase that needs a heads-up beyond the routine welld bounce.

Sidebar: Could we just raise the SSH timeout?

Tempting, and wrong. A longer SSH-ready gate means a saturated box takes even longer to tell you it's in trouble, and the timed-out creates pile up holding real VM resources the whole time. The gate isn't too short — the box is too busy. Fix the busy, not the patience.

Part VThe Part of Tens

Chapter 9: Ten Things Admission Control Fixes (and Three It Doesn't)

FIXES:

  1. Bake-under-load timeouts — the May 14 incident, gone.
  2. The "destroys first" lecture — no longer needed; the box paces itself.
  3. Orphan VZ processes from timed-out creates — fewer failed creates means fewer orphans to reap.
  4. Resurrection-burst stampedes — a welld bounce no longer slams a dozen boots at the host at once.
  5. Operator-vs-cells contention — a human running well exec can't accidentally starve a cells bake.
  6. Unpredictable create latency — bounded concurrency means bounded, explainable wait times.
  7. "Is wells broken?" false alarms — pacing is visible and expected, not mysterious.
  8. Wasted ~5-minute failures — a boot that would time out now waits cheaply instead of failing expensively.
  9. Cross-team chat-channel debugging — fewer "what's going on with the substrate?" threads.
  10. Future multi-host scaling pain — per-host admission control is the natural unit when a Colony spans Macs.

DOESN'T FIX:

  1. A genuinely undersized host — if you need 100 wells on a box that fits 20, admission control just makes the wait honest. It paces; it doesn't conjure cores.
  2. A slow base image — if cloud-init is slow on an idle box, that's an image problem, not an admission problem.
  3. Bugs inside the guest — wells watches the box from outside. What the agent does inside the VM is still cells's domain.

Chapter 10: Ten Questions the Cells Team Will Ask

  1. "Does our code change?" — No. The REST contract is identical.
  2. "Will create get slower?" — Only when the box is already saturated, and only instead of failing. On an idle box, no change.
  3. "What's the default limit?" — A flat 3 to start, then a load-aware ratio (~1.5–2× cores). Both are env-tunable.
  4. "Will our HTTP client time out waiting for a slot?" — The real open question. Either we bound the wait or we surface a "queued" signal. We want cells's input on the create-call timeout budget.
  5. "Can we see when we're being throttled?" — Yes, in Phase 4: well doctor and /healthz will show admission depth.
  6. "Does exec count toward the load?" — Open question. exec burns CPU but isn't a boot. Probably it should influence the limit without holding a slot. Wants discussion.
  7. "Can we opt out for a priority bake?" — Not in scope. No priorities (Chapter 7). If this turns out to matter, it's a separate proposal.
  8. "Does this replace our pool refill logic?" — No. Different axis (Chapter 6). Your pool logic is untouched.
  9. "What happens to in-flight boots if welld bounces?" — They keep running — they hold real OS resources, not semaphore tokens. The semaphore just resets empty.
  10. "When can we have it?" — Phase 1 is ~a day once approved. It's small on purpose.

Where to Go From Here

This proposal is deliberately small. It is one new lib/ module, three call sites, and an env knob — and it removes an entire class of "the substrate fell over under load" incidents, plus the manual guidance we keep handing the cells team to avoid them.

Where it landed, in order:

  1. Done — Pete approved + built. Phases 1–4 shipped to main (63a3b94): the BootGate in lib/admission.ts, wired into createWell / startWell / wakeWell, with boot_gate on /healthz. 14 unit tests; suite 1001 → 1015.
  2. Goes live on the next welld bounce — coordinated with cells, not surprise-bounced.
  3. Cells weighs in on the two open questions in Chapter 10 — the create-call timeout budget, and whether exec should influence the limit. Refinements, not blockers.
  4. The companion skill. The wells Claude Code skill gets a "load & ordering" section — see the Remember box in Chapter 2. Primitive and skill, not primitive alone.
