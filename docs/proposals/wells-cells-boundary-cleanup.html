Wells / Cells boundary cleanup

Status: Proposal · Date: 2026-05-12 · From: wells team · To: cells team

This proposal is a follow-up to the cells-pool-on-wells.md hand-off (2026-05-09) and the W.68 / W.70 incident at 09:01Z today. The hand-off proposed a boundary; the code didn't move; the incident is what changed the cost calculus.

TL;DR

  • Wells today owns both VM lifecycle primitives (its job) AND cells's pool / egg orchestration (scope creep from Phase A.1.4).
  • The W.68 incident was a layered-abstraction leak — wells trying to enforce a cells-shaped invariant ("running cell is reachable") and reaching into substrate plumbing it shouldn't write to.
  • Propose: wells = VM primitives only. Cells owns the pool, egg lifecycle, identity reset, birth orchestration. Three slices, smallest first.

Why now

The W.68 incident, 2026-05-11 09:01Z

Cells team reported 4 of 5 running cells went dark within minutes of the W.68 deploy. lume reported running; ping/SSH timed out. Root cause: welld's lease publisher was kicking macOS bootpd ~96 times per minute (16 alive wells × 6 sweeps/min, each publish triggered a SIGKILL+restart of bootpd). In-flight DHCP renewals failed mid-handshake; guests lost their leases on the vmnet bridge.

Mitigated in 5 minutes (sweep disabled, then batched-kick fix shipped as W.70). But the deeper question: why was welld writing to /var/db/dhcpd_leases at all?

Wells was reaching into substrate plumbing to enforce an invariant — "if welld says status=running, the cell is reachable" — that's shaped by cells's needs, not the substrate's. Wells doesn't care if a cell is reachable; cells does. By holding the invariant, wells got entangled with cells's failure modes.

The pool is the same pattern at a larger scale. ~/.wells/pool/ exists solely for cells's Tier 4 birth wedge. Wells maintains it because welld is the only daemon — but the purpose is cells-shaped, and the orchestration code in wells is hard to evolve without coordinating with cells's bake flow.

Current state — where the layers leak

Current layering — wells holds ~1100-1300 LoC of cells-shaped orchestration.

Proposed state — clean separation

Proposed layering — wells is substrate primitives only; cells composes them into pools, eggs, and birth flow.

Who owns what

wells substrate

  • Linux VM lifecycle on macOS (lume, VZ, vmnet)
  • Hibernate / wake primitives (Apple VZ saveState/restoreState)
  • DHCP observation — wells reads /var/db/dhcpd_leases for IP lookups but doesn't write to it (post-cleanup)
  • Image store + R2 push/pull
  • REST API + well CLI
  • Engine pluralism (vwell wrapper, future Firecracker/QEMU support)
  • Per-well registry + state machine
  • Reports honest observed state: "running + ip: null" is a valid response — wells doesn't try to enforce reachability

cells agent

  • Bake recipe — install DNA, agent stack (pi, bun, claude-code), cell user
  • Pool of pre-baked hibernated wells — state in ~/.cells/pool/
  • Pool refill loop — calls well create + bake + well hibernate
  • Birth orchestration — well wake + identity rotation via SSH (hostname, machine-id, DNA refresh)
  • Reachability recovery — if their cell is unreachable, decide stop+start vs destroy+recreate vs wait
  • Host-bridge architecture, talk path, mother→cell protocol
  • Tier 4 birth wedge — running-resident eggs

How a birth flow looks today vs proposed

Today (mixed responsibility)

  • Cells calls POST /v1/wells with from_image: cell-base
  • Wells's handleCreateWell tries pool adoption first; if pool is empty, falls through to fresh-create
  • If pool hit: wells does adoptFromPool — renames bundle, runs identityReset.ts over SSH (hostname, machine-id)
  • Cells then runs its OWN bake-style steps via additional well exec calls (DNA install if needed)
  • Background: welld's pool filler bakes the next member, runs its own warming sequence, hibernates

Two identity rotations (wells's reset + cells's bake) sometimes overlap. Pool's bake recipe lives in welld; cells's bake recipe lives in cells. Drift risk.

Proposed (clean ownership)

  • Cells's pool manager pre-bakes hibernated wells: POST /v1/wells + cells's bake + POST /v1/wells/<n>/hibernate
  • Cells's birth flow: pick a hibernated egg from cells's pool
  • Cells calls POST /v1/wells/<n>/wake — wells does VZ restore
  • Cells SSHes via well exec + does their identity rotation (just like their existing bake's identity-setup path)
  • Cell is alive at <n>.cells.md via vhost dispatch — done

One bake recipe (cells's). One identity rotation (cells's). Wells stays out of cells's identity decisions.

Migration — three slices

Slice 1 — no coord needed
Drop the lease publisher
~250 LoC delete · ~30 min

Remove W.68's active lease enforcement entirely. Keep release-hostname for destroy cleanup, keep orphan-only /flush (W.67). Wells observes /var/db/dhcpd_leases, doesn't write to it outside destroy.

Impact on cells: none — publisher was already disabled post-incident. Reachability recovery (the original W.68 motivation) becomes cells's job, which it already kind of was.

Slice 2 — coord required
Move pool to cells
~1100 LoC delete from wells · ~2-3 days cells-side

Cells implements pool management on top of wells's REST primitives. State migrates from ~/.wells/pool/ to ~/.cells/pool/ (or wherever cells wants). Wells deletes poolFill.ts, poolFiller.ts, poolRegistry.ts, adoptFromPool.ts, well pool CLI.

Impact on cells: medium — port pool refill logic. cells pool refill calls POST /v1/wells + bake + hibernate. Cells gains iteration speed on the bake recipe (no welld coordination).

Slice 3 — small follow-on
Simplify createWell warming
~100 LoC simplify · ~30 min

Post-slice-2, fresh-create wells don't need to be hibernate-legal by default. Drop the boot-with-cidata → stop → restart-without-mount → second-DHCP warming sequence. Operator wells boot with cidata and run as alive_running.

Impact on cells: none — cells's pool already detaches cidata in its bake before hibernate. Fresh-create just gets ~6-8s faster (no second-boot wait).

Recommendation

Ship slice 1 immediately. No coordination needed, eliminates the W.68-class bug surface, ~30 min of wells-side work. Publisher is already disabled — slice 1 just deletes the code.

Schedule slice 2 when cells has bandwidth. The 2026-05-09 hand-off proposed this; the W.68 incident showed why it matters. ~2-3 days of cells-side work, ~half-day of wells-side delete.

Slice 3 follows slice 2 naturally. Wells's createWell gets simpler once the warming-for-pool-hibernate requirement is gone.
