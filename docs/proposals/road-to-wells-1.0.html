In plain English

For anyone who doesn't live in this codebase

Wells is the part that runs Linux VMs on a Mac. The "apartment building," in this project's language. Cells (the agent code) lives inside those VMs. We've been shipping wells for about a week — the MVP went out 2026-05-06, and just this morning we squashed everything that's gone in since then to v0.2.0. A bunch of operational maturity, a much faster start-up path, image management, all the unglamorous stuff that turns "it works on my machine" into "it works under real load."

1.0 isn't a feature. It's a state of being. It means: wells is stable enough that the cells team can build their actual product on top of it without expecting it to wobble. It means we've cleaned up the messy boundary between wells's work and cells's work (that's the sprint that's running right now). It means a second person could install wells on their Mac and have it work. And it means we've finished the small handful of items from our Phase A plan that haven't been ticked yet.

What's left, plainly:

  • This week: deploy the static-IP fix (kills a whole class of incidents), publish the new image alias (lets cells team stop pinning to a specific bake), delete some dangerous old code.
  • Next week: cells team moves their pool-of-pre-baked-cells code into their own codebase. We don't touch the substrate during that — it stays stable so they can work against it.
  • Week after: we delete the ~1,100 lines of pool code from wells (cells owns it now), and close out two small Phase A boxes that have been sitting open.
  • Early June: rename the project folder + GitHub repo from splites to wells (it's been called wells everywhere except those two places for months). Polish install docs so a second operator could set it up. Tag v1.0.0.

What we're explicitly not doing in 1.0: multi-Mac scaling, dynamic memory packing, support for non-Linux guests, cloud hosting. All of those are real ambitions captured in the roadmap — they're just 1.x, not 1.0. The 1.0 bar is "stable substrate for one operator's local fleet," not "every feature ever proposed."

Why this is realistic: most of the hard work landed already. The hibernation/wake plumbing, the lume-process supervisor, the warm pool, the image library on R2, the cells-shape API — all shipped. What's left is the cleanup + the items that have been deliberately deferred because cells team needed our substrate quiet, not changing.

4Scope

What's in 1.0 and what's 1.x

In 1.0

  • Phase A — autosleep, warm pool, R2 sync, retention (all shipped + A.1.3.c/g residuals)
  • Phase B wells-side — lifecycle truth, hibernate/wake, fork hardening, image contract (all shipped)
  • Boundary cleanup sprint — static IPs, image alias, pool migration, ~1,100 LoC removed
  • splites → wells rename (folder + GH)
  • Multi-operator install docs
  • Cells V1 acceptance fully green against the substrate
  • Architecture + state-schema docs current

Not in 1.0 (1.x and beyond)

  • A.2 Frozen tier — R2 hibernation offload. Stretch goal (could slide in if cells consumer surfaces). Currently no concrete need; defers cleanly.
  • A.3 Egress enforcement — explicitly deferred 2026-05-11, no consumer.
  • Phase C — memory chunks. Needs real-load data first; 1.x.
  • Phase D — multi-Lab Colony. Depends on A.2 Frozen tier; 1.x or later.
  • Phase E — cloud hosting. Deprioritized indefinitely.
  • Multi-OS guests (macOS / Windows / Android via VZ + QEMU).
  • GPU passthrough.
  • Engine swap to Apple containerization.

7After 1.0

What 1.x looks like, so 1.0 stays honest

1.0 means substrate-stable, not feature-complete. The roadmap (see docs/ROADMAP.md) keeps three named ambitions for 1.x:

  • v1.1 — A.2 Frozen tier (if it slips out of 1.0). R2 hibernation offload. Unblocks Phase D cell migration. ~1 week.
  • v1.2 — Phase C memory chunks. Lume balloon control + pressure controller. Lets us pack 2-3× more cells per Mac. Requires real-workload data from Phase B operations to tune. ~2 weeks.
  • v1.3 / v2 — Phase D multi-Lab Colony. Multiple Macs as one logical fleet. Depends on Frozen tier. ~2 weeks once C is shipped.

The "Future" bucket in ROADMAP (multi-OS guests, GPU passthrough, named volumes, containerization swap) stays explicitly unplanned — drops into MVP-PLAN when there's a real consumer asking for it.
