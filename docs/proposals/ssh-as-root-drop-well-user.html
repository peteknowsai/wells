SSH as root, drop the well user

Status: proposal — needs cells team sign-off before scheduling  ·  Author: wells  ·  Date: 2026-05-22  ·  Follow-on to: exec-default-root (shipped wells main 6488eaf)

TL;DR

  • Now that exec/console default to root, the well user is no longer an exec target — it survives only as the SSH transport identity (we ssh well@ip, then sudo -H -u root).
  • Proposal: lay the operator key into /root/.ssh/authorized_keys, ssh root@ip directly, and delete the well user entirely — the useradd, the 90-well sudoers file, the authorized_keys install.
  • Removes one user, one sudoers file, and the sudo hop on every exec/console/WS call. The default path becomes a plain ssh root@ip -- cmd.
  • Cost: it's a well-firstboot.sh change → base-image rebake → cells must rebake cell-base. That's the whole reason this is a proposal, not a shipped change.
  • Not urgent. exec-default-root already closed the bug class. This is pure simplification.

In plain English

What this is, without the jargon

Every well (Linux VM) has a few login accounts. When you run a command in a well, wells logs in over SSH as a helper account called well, then immediately switches to the root account to actually run your command. Two steps where one would do.

The well account only ever existed because, historically, commands defaulted to running as a non-root user and we needed a known SSH door to come in through. We just changed the default so commands run as root — so the well account is now a doorway to a room nobody stands in.

This proposes giving root its own SSH door and demolishing the well account. Fewer accounts, fewer config files, one less step on every command. The only catch: the account list is baked into the VM template image, so removing it means rebuilding that image — and the cells team builds their own image on top of ours, so they'd need to rebuild theirs too. Hence: we ask first.

Why now

The well user had one job: be the SSH entry point so well exec could land somewhere and then sudo-switch to whatever --user named. With the exec default at well, the common case was "land as well, run directly, no sudo." That made well load-bearing.

As of 6488eaf the exec default is root. The common case is now "land as well, sudo -H -u root, run." The well user is pure overhead on the hot path — it is never itself the thing you want to run as. A user that exists only to be sudo'd away from is a user that should not exist.

Current state — three SSH identities

For clarity, wells guests currently carry three host-reachable accounts. This proposal touches exactly one.

UserHomeWho SSHes as itThis proposal
well/home/wellwell exec / console / HTTP+WS exec — the exec transportdeleted
ubuntu/home/ubuntuwelld-internal flows: create/warm probes, checkpoints, image validation, services, lifecycleuntouched
cell/cellcells host-bridge (ssh cell@ip direct); cells's agent useruntouched

Only the exec transport moves. The ubuntu internal-ops identity and the cell agent identity are out of scope.

Proposed change

1. firstboot — templates/well-firstboot.sh

Today (lines 96–104) firstboot installs authorized_keys for ubuntu, then useradds well, installs its authorized_keys, and writes /etc/sudoers.d/90-well. Replace the well block with a root key install:

# Operator key → root. Ubuntu's sshd default is
# `PermitRootLogin prohibit-password`, which permits key-based
# root login and refuses passwords — exactly what we want.
install -d -o root -g root -m 0700 /root/.ssh
install -o root -g root -m 0600 "$SEED/authorized_keys" /root/.ssh/authorized_keys

Delete: the useradd … well, the well authorized_keys install, and /etc/sudoers.d/90-well. The cell key install (lines 111–114) and the ubuntu install stay. Optionally add an explicit PermitRootLogin prohibit-password to the existing sshd_config.d drop-in to pin the behavior rather than rely on the distro default.

2. Host-side SSH targets

well@ becomes root@, and the sudo wrap drops out. Touch points:

FileChange
cli/well.ts — cmdExecwell@ip → root@ip; drop the sudo -n -H -u wrap (keep it only for non-root --user)
cli/well.ts — cmdConsolewell@ip → root@ip; drop the sudo -i for the default case
daemon/welld.ts — WS execwell@ip → root@ip; ensureSshMaster user → root
daemon/welld.ts — runHttpExecSshsame
lib/sshControl.tscontrol-master default user well → root
lib/createWell.ts:390well@<name> warm probe → root@<name>

The sudo-switch logic stays for explicit --user overrides — --user cell, --user ubuntu still SSH as root and sudo -H -u <target>. Only the default case loses the hop. (Note: the pivot user becomes root, so the sudoers requirement flips from "well can sudo to anyone" to "root is root" — trivially satisfied.)

3. Docs

architecture.md users section, cells-integration.md SSH section — drop well from the user list, note root is the SSH entry identity.

Rollout

  • 1. Land the code change behind the rebake. The host-side root@ switch and the firstboot edit ship together — a fork from the new base image has a root key but no well user; a fork from the old image has a well user but no root key. They must move as one. Old wells already running keep working until re-forked.
  • 2. wells rebakes ubuntu-25.10-base with the new firstboot.
  • 3. cells rebakes cell-base on top of the new base to inherit the root key install. This is the cells action item.
  • 4. Verify — well exec on a fork lands as root with no sudo hop; --user cell / --user ubuntu still work via the retained sudo path.

Risks

RiskAssessment
Root SSH login refusedLow. Ubuntu's prohibit-password default permits key-based root login. We pin it with an explicit drop-in.
In-flight wells on old imageNone. Already-forked wells keep their well user and keep working; the change only affects forks from the new base. Mixed fleet is fine during the transition.
cells host-bridge breakageNone expected — the bridge uses cell@ and ubuntu@, neither of which moves. Cells to confirm nothing dials well@ directly.
Something runs as well inside the guestCells to confirm — no service, cron, or DNA path expects /home/well to exist.

Alternative: do nothing

Entirely defensible. The well user costs one useradd, one sudoers file, and one sudo exec per call — all cheap. exec-default-root already killed the bug class this is adjacent to. If cells would rather not absorb a rebake for a cosmetic simplification, "keep well as the transport user forever" is a fine answer and this proposal goes in a drawer.

Questions for the cells team

1. Does anything cells-side SSH or connect to well@<ip> directly, or expect /home/well to exist inside the guest?

2. Is a coordinated cell-base rebake worth it for this simplification — or would you rather we keep the well transport user and not touch the base image?

3. If yes: any timing constraint? We'd land the code change paired with the base rebake and ping you to rebake cell-base right after.
