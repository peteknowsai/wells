Internal pitch · post-1.0 quarter

Wells 2.0

The next 12 weeks. What we build now that the substrate works.
Drafted 2026-05-13 · One-author shop with AI on speed-dial · Compressed timeframe: what would have been 3-6 months in 2023 is 8-12 weeks now.

TL;DR

  1. The substrate works. 1.0 is shippable. Hibernate, wake, seal, pool — all green. Cells team is consuming it daily. Now we make it livable.
  2. Three themes for the quarter: Always-On, See Everything, Reach Outward. In that order. The dashboard is the highest-leverage thing on the list — visibility is what unlocks daily-driver confidence.
  3. The bet: by August, Pete checks wells.cells.md from his phone and sees his fleet running. The Mac never reboots without wells coming back. The cells-on-top story has somewhere to land.

Where we are

SHIPPED Phase A (operational maturity) · Boundary cleanup (Pi 1-3 + /seal) · 989 tests green sequential · 4 clean welld bounces during the boundary arc · Cells's V1.5 sleep 589ms, V1.10 wake 380ms on /seal-baked members.

In plain English: Wells is a daemon on your Mac that runs Linux boxes. You can make one, talk to it, freeze its RAM to disk, thaw it again. Cells (the agent framework) sits on top and uses wells as the body for each agent. The plumbing is done. Now the question is: how do we make this thing pleasant to actually live with?

What's missing — and what this pitch is about:

  • Wells doesn't auto-start on boot. If Pete restarts the Mac, he runs bun run welld by hand.
  • There's no UI. No place to glance and see "11 wells, 3 alive, 1 hibernating, host has 60% RAM free."
  • Wells only listens on 127.0.0.1:7878. Pete can't check on it from his phone at a hot spring.
  • If welld crashes, there's no supervisor. If lume serve dies, we restart it — but if welld dies, the fleet goes dark until Pete notices.

The substrate is solid. The experience is a 0.4. We fix that in this quarter.

The strategic bet

Three themes, sequenced for compounding leverage:

  1. Always-On — the daemon survives reboots and crashes. Foundation for everything else.
  2. See Everything — local web dashboard. Pete can glance at http://localhost:7878 and know the state of the world. This is the unlock for daily-driver confidence.
  3. Reach Outward — wells.cells.md serves the same dashboard from anywhere. Plus the foundations for external wake-on-traffic (the per-well Worker layer from the original roadmap).

Why this order: visibility multiplies confidence. Confidence multiplies usage. Usage exposes the next set of problems. Build the feedback loop before scaling the scope.

Don't build features for users you don't have. Build the dashboard you'd want, ship it to yourself, dogfood for two weeks, then decide what's next.

Theme 1 — Always-On WEEKS 1-2

The goal

Wells is part of the Mac. You don't think about starting it any more than you think about starting Finder.

In plain English: Right now if Pete reboots his Mac, the wells daemon is dead until he opens a terminal and runs it manually. The Mac doesn't know wells exists. We fix that by registering wells with macOS's startup system (launchd), telling it to restart wells if it crashes, and making the recovery clean enough that Pete never has to think about it.

What we ship

PieceWhat it does
well daemon installOne command installs the launchd plist. Wells starts on login. Survives reboot. KeepAlive auto-restarts on crash.
well daemon statusShows: launchd state, uptime, last crash (if any), lume serve subprocess pid, version, registry size.
well healthSelf-check: state-dir perms, lume reachable, port free, tunnels alive, disk space, RAM headroom. Returns 0/non-zero. Cron-friendly.
Crash diaryEvery welld crash appends to ~/.wells/crash.log with a stack and the last 100 log lines. Surfaced in daemon status.
Wells-resume policyStrong opinion: wells that were alive_running at shutdown come back alive_running after reboot. Wells that were hibernating stay hibernating. No surprises.

What we're already partway through

scripts/welld.plist.template + scripts/install-launchd.sh exist but aren't promoted into the CLI. We polish, document, and make them the default install path. docs/install.md § 9 already covers the launchd story — we operationalize it.

Acceptance

  • Pete reboots his Mac. He logs in. Within 15s, well list works without him doing anything.
  • kill -9 $(pgrep -f welld) — welld restarts within 5s, registry intact, running wells reattached.
  • A pre-reboot snapshot of running wells matches the post-reboot snapshot (modulo wake latency).

Theme 2 — See Everything WEEKS 3-5

The goal

One URL. Pete opens it. He knows what's happening.

In plain English: The most-asked question Pete will have, every day, is "what's running right now?" Today the answer is well list in a terminal. That's fine for inspection, terrible for confidence. The dashboard makes the answer ambient: just glance at it. Wells alive, wells hibernating, host pressure, recent events. That's it. No analytics, no graphs over time (yet), no settings page (yet). Just the truth, right now.

What we ship

A single-page web app served by welld itself at http://localhost:7878 (root). Server-rendered HTML + minimal vanilla JS for polling. Zero build step, matches the no-bundler stack rule.

Pages

  • Overview (the screenshot above). Roster + host pressure + recent events. Polls every 2s.
  • Well detail — click a well, get its full state: meta, runtime, recent transitions, last 200 log lines (lume + welld). Tail mode.
  • Events log — chronological wells-wide event stream. Filter by well, type, severity.
  • Host — disk usage in ~/.wells/ + ~/.lume/, lume serve status, vmnet leases, IP pool reservations.

What it's NOT (this quarter)

  • No login. Local-only by default; bearer-protected when exposed externally (Theme 3).
  • No graphs over time. We're not building Grafana. Surface state, not trends.
  • No mutating actions in the UI. (Read-only this quarter. Buttons come later, after we know what we'd want.)
  • No SPA framework. HTML + a sprinkle of JS for the polling loop.

Why this is the highest-leverage piece

Every other improvement compounds against the dashboard. Notice a wake taking 2s instead of 380ms? Open the dashboard, see it. Wonder why scratch hibernated? Check the event log. Curious if cells's pool is keeping up? Glance at the pool column. The dashboard becomes the answer to "is wells okay?" — which Pete asks himself five times a day.

Acceptance

  • Pete opens the dashboard. Within 1 second he knows: how many wells, what state, is the host stressed.
  • Dashboard works offline (no CDN dependencies, all assets served by welld).
  • Dashboard survives welld bounces without browser refresh (graceful "reconnecting…" banner).

Theme 3 — Reach Outward WEEKS 6-9

The goal

https://wells.cells.md serves the same dashboard. From Pete's phone, from a coffee shop, from anywhere. Wake-on-traffic from outside the LAN.

In plain English: Right now wells lives inside Pete's Mac. The dashboard from Theme 2 will only work on Pete's Mac. We extend it: a public URL (wells.cells.md) that proxies through Cloudflare to a tunnel running on the Mac, talks to welld, and serves the same dashboard. Same view, anywhere. Plus: external traffic to a specific well (like pete.wells.cells.md) hits a hibernating well → it wakes → traffic flows. The full "fleet on your closet shelf" experience.

What we ship

PieceWhat it does
CF Tunnel for welldcloudflared registered as a launchd service. Maps wells.cells.md → localhost:7878. Bearer-auth at the welld layer (separate from local token).
External authSecond bearer (WELL_EXTERNAL_TOKEN) for the public surface. Stored in 1Password, never in the repo.
Per-well subdomains<name>.wells.cells.md → guest:8080 via welld's reverse proxy. Already half-built — finish it.
Wake-on-external-trafficTraffic to <name>.wells.cells.md when the well is hibernating triggers wake + buffers the request. Cell team has the patterns; we plumb the wells side.
Mobile-first dashboard polishThe Theme 2 dashboard already mobile-friendly. Here we add iOS shortcuts: tap → wake well X, tap → snapshot well Y.

The Worker layer — deferred but plumbed

The original roadmap calls for each well to have its own CF Worker + Durable Object as a wake-on-demand edge. That's real external-reach architecture — it lets a phone tap traffic-wake a well in <200ms even if the daemon is hibernating its own watchdog. We don't build the Worker layer in this quarter, but we make sure the welld interface won't have to change when we do. Specifically: welld grows a POST /v1/wells/<n>/wake-token endpoint that issues a short-lived signed wake key, which the Worker will eventually consume.

Acceptance

  • From Pete's phone, on cellular, wells.cells.md shows the same dashboard as localhost.
  • Tapping a hibernating well's URL wakes it. End-to-end <3s including DNS + TLS + tunnel.
  • Tunnel survives Mac sleep/wake without manual intervention.
  • Bearer auth fails closed if WELL_EXTERNAL_TOKEN isn't set — no accidental open dashboard.

Theme 4 — The Distance Beyond STRETCH · WEEKS 10-12

If themes 1-3 land cleanly with time to spare, pull one of these forward:

4a. Egress policy enforcement (the A.3 deferred ticket)

Wells claims to have egress policy (POST /v1/wells/<n>/policy/network) but it's a stub. Real pf-rule teeth. Becomes load-bearing the moment we run untrusted agents inside cells — which is sooner than we think.

4b. Linux engine spike (Firecracker on a NUC)

Prove the multi-engine architecture by running a single well via Firecracker on a Linux box. Same API. Same dashboard view. Same cells-side experience. No production traffic — just the proof. Unblocks the eventual "host wells on a VPS" or "host wells on a closet NUC" stories.

4c. Frozen tier — R2 hibernation offload (A.2)

Hibernate state → R2 for cells we won't touch for weeks. The Multi-Lab Colony (Phase D) depends on this. We don't build the colony — we build the offload pipe so the colony has somewhere to land.

4d. Multi-host single-Colony spike

Two Mac Minis acting as one Colony. wells API on each, but a unified roster. Pete's been hinting at this since the original naming doc. Worth a week of spike-it-or-kill-it.

Recommended pick: 4a. Egress is the only one that's load-bearing on near-term cells usage. The rest are great but speculative until cells exercises them.
