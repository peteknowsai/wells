1V1.3 first-token target: keep 5s, or bump to 8s?

Cells team is asking. Affects their perf SLA, not our substrate.

In plain English

When someone starts a new cell and asks it something, how long until the first letter of the answer appears on screen? Today we promise 5 seconds. Measuring shows we hit about 7.3 seconds in practice. Cells team is asking: is that actually OK? They argue: the birth animation runs 5 seconds anyway, so the user doesn't notice the extra 2 seconds — it's hidden by the loading state. So instead of trying to make the substrate faster (which would mean architectural changes), just call it a 8-second target and move on.

Technical context

The architectural cost is the host-bridge layer added in V1.0 — a fresh SSH session + a fresh pi spawn per cell talk. That fixed the V1.0 cell-side bridge issues but cost ~2s of latency per first-talk. Cells team considered the trade fair at V1.0 ship time and re-validated during acceptance.

Writeup: ~/Projects/cells/docs/perf/birth-to-greeting.md.

Options

Bump target to 8s cells's recommendation

Acknowledges the architectural cost is real and gives them ~30% headroom over today's measured p50. UX absorbs it via the animation.

Keep 5s and tell cells to chase it

Forces them to invest in pi-process-reuse or session-pooling to claw the 2s back. Real engineering work, weeks. Probably right when total volume is much higher; arguably premature now.

Compromise — 6.5s

Splits the difference: acknowledges the cost but keeps some pressure on the perf budget.

What's needed from you

Just an answer in the chat channel ("8s ok" works). Cells team updates their docs + status board.

3Bake mode: overwrite, or new tag + alias swap?

Now that we have the alias system, we have a choice we didn't have before.

In plain English

To activate static IPs, we have to rebuild the foundational Ubuntu disk image with a new firstboot script. The image lives at ~/.wells/images/ubuntu-25.10-base/disk.img.

Choice: overwrite that file in place, OR bake to a new name (ubuntu-25.10-base-w72), verify it works, then flip the ubuntu-base alias to point at the new one (atomic, reversible).

The second option is safer because if something goes wrong we just unflip the alias and we're back. Now that we have aliases, this is essentially free.

Technical context

Today's scripts/bake-base-image.ts overwrites ubuntu-25.10-base when --force is passed. The new behavior (committed today) is that after a successful bake the script auto-points ubuntu-base at ubuntu-25.10-base. To get the safer flow, I'd add an option to bake to a versioned name (ubuntu-25.10-base-w72) and skip the auto-alias — then a separate operator command flips the alias once verified.

Options

New tag + manual alias flip recommended

Bake to ubuntu-25.10-base-w72. Live-verify by creating a well with --from-image=ubuntu-25.10-base-w72. When green, run well image alias ubuntu-base ubuntu-25.10-base-w72. Rollback = re-flip to the older version. Costs ~20 min of my time to add the bake-script flag + a small CLI command.

Overwrite in place

What the script does today with --force. Simpler, but if the bake fails or the new firstboot has a bug, the base is half-rotten until someone re-bakes. Rollback path: re-bake again with a fix. ~15 min wall-clock to recover.

What's needed from you

Pick the bake mode. If you pick the recommended one, I add the safety hooks before the deploy day; if you pick overwrite, no code change needed.

4Default static IP range: turn it on now, or via operator config?

Smaller call. Affects what fresh installs do out of the box.

In plain English

Right now the static IP feature is built but turned off by default (a setting called static_ip_range defaults to "null"). When we deploy, we'll turn it on for our host. Question: should the hardcoded default be "on" (range 200-250) for any fresh wells install, or should every operator opt in by editing their settings file?

Technical context

Hardcoded default in lib/defaults.ts is "static_ip_range": null today — operator opts in. If we ship it as "200-250", every new wells install with a re-baked base gets static IPs without thinking about it. The downside: anyone who has bootpd configured to grant in .200-.250 hits our startup overlap check and welld refuses to start with a clear error. They'd then either narrow bootpd or set static_ip_range to a different range. Either way, no data loss; just a friction moment.

Options

Keep default null (operator opts in) recommended

Defensive — no surprises for anyone running on a non-default vmnet config. We turn it on for our host explicitly via defaults.json. Downstream operators see the feature in docs and opt in when ready.

Flip the hardcoded default to "200-250"

Eager adoption — every fresh wells install gets the benefit. Tradeoff: the few operators on weird vmnet configs see a startup refusal (with a clear error pointing at the fix) on their next wells upgrade.

What's needed from you

A nod. I either keep the hardcoded default at null and we set ours explicitly, or I flip the default in a one-line code change.
