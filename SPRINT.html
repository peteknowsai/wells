In plain English

For anyone who doesn't live in this codebase

Tonight at 09:01 we had a substrate incident — four of cells's running cells went dark when wells's code interfered with the Mac's network plumbing. We patched it in five minutes, then realized the root cause is deeper: wells and cells have been sharing code responsibilities in a way that makes these incidents predictable. The boundary between "what wells does" and "what cells does" has been blurry for months.

This sprint cleans that up. Going forward: wells builds and runs Linux VMs on the Mac (the "apartment"). Cells decides what goes inside those VMs — the AI agent code, the pool of pre-baked cells ready to use, the identity rotation that makes each cell unique (the "interior decoration and tenants").

The work splits naturally into three phases:

  • This week (wells): One real architectural change (stop asking the Mac's DHCP for IP addresses — wells picks them instead, removes the cause of tonight's incident class). Two small follow-ups (delete dead code, add a stable image name).
  • Next week (cells): Cells team takes ownership of the "pool" code — about 1,100 lines moving from wells into cells. They're doing this work; we're in support mode.
  • Week after (wells): We delete the old pool code from wells. Substrate becomes ~40% smaller, ~40% less surface for bugs.

Why we're confident this works: the boundary we're proposing was already written down on May 9 in a "cells-pool-on-wells.md" hand-off doc. We just never moved the code. Tonight's incident is what made it worth the effort to actually do the move. Both teams agreed on the plan in chat this morning; we're locked in.

What "done" looks like for this sprint: Wells is smaller. Wells doesn't write to macOS DHCP files. Cells owns the pool. The substrate is sturdier than it was before tonight — by construction, not by patching.
